# DBC 파일 관리 전략 가이드

## 전략 비교: 선별 추출 vs 전체 레퍼런스 사용

### 🎯 현재 전략: 선별 추출 방식 (권장 ✅)

**방법**: Hyundai/Kia 레퍼런스 DBC에서 **필요한 신호만 선별**하여 프로젝트 자체 DBC와 통합

#### ✅ 장점

1. **성능 최적화**
   - CANoe 로딩 시간 단축 (1,677줄 → 251줄, 85% 감소)
   - 시뮬레이션 실시간 성능 향상
   - 메모리 사용량 감소

2. **유지보수성**
   - 프로젝트에 **실제로 사용하는 신호만 관리**
   - DBC 파일 가독성 향상 (17개 메시지 vs 100개 이상)
   - 변경 추적 용이 (Git diff가 깔끔)

3. **프로젝트 명확성**
   - 아키텍처와 DBC 1:1 매핑 가능
   - 요구사항 추적성 향상
   - 불필요한 의존성 제거

4. **버전 관리**
   - 프로젝트 독립성 확보
   - Hyundai/Kia 레퍼런스 업데이트와 무관하게 운영 가능
   - 필요 시 선택적으로 레퍼런스 업데이트 반영

5. **테스트 효율성**
   - Fault Injection 시나리오 간소화
   - 검증 범위 명확화
   - 테스트 케이스 작성 용이

#### ⚠️ 단점

1. **초기 작업량**
   - 필요한 신호 식별 및 추출 작업 필요
   - 레퍼런스와 커스텀 DBC 통합 작업

2. **누락 위험**
   - 나중에 필요한 신호를 빠뜨릴 수 있음
   - **대응**: 리뷰 리포트와 같은 정기적인 검증

---

### 🔄 대안: 전체 레퍼런스 사용 방식

**방법**: Hyundai/Kia 레퍼런스 DBC 전체를 사용하고, 커스텀 신호만 추가

#### ✅ 장점

1. **완전성**
   - 모든 Hyundai/Kia 신호 사용 가능
   - 나중에 필요한 신호 누락 걱정 없음

2. **초기 작업 간소화**
   - 선별 작업 불필요
   - 바로 레퍼런스 사용

#### ❌ 단점

1. **성능 문제**
   - CANoe 로딩 시간 증가 (1,677줄 전체 파싱)
   - 시뮬레이션 실시간 성능 저하
   - 메모리 낭비

2. **유지보수 어려움**
   - DBC 파일 가독성 저하 (100개 이상 메시지)
   - 프로젝트와 무관한 신호들로 인한 혼란
   - Git diff 추적 어려움

3. **프로젝트 불명확성**
   - 실제로 사용하는 신호 vs 사용하지 않는 신호 구분 어려움
   - 아키텍처와 DBC 간 매핑 복잡
   - 요구사항 추적성 저하

4. **테스트 복잡도**
   - Fault Injection 시나리오 과다
   - 검증 범위 불명확
   - 불필요한 테스트 케이스 증가

## 📊 현재 프로젝트 DBC 현황

### `vehicle_system.dbc` (통합 DBC)
- **메시지 수**: 17개
- **신호 수**: ~100개
- **크기**: 251줄
- **출처**:
  - Hyundai/Kia 레퍼런스에서 선별 추출 (12개 메시지)
  - 프로젝트 자체 제작 (5개 메시지: ADAS, IVI)

## 🎯 최종 권장사항

### ✅ **선별 추출 방식 유지 (현재 전략)**

**이유**:
1. **IVI vECU 프로젝트 특성상 필요한 신호만 명확히 정의됨**
   - 조명 제어, 안전 경고, ADAS 연계
   - 불필요한 파워트레인/섀시 신호 다수 (LPI, REA, FPCM, OPI 등)

2. **CANoe 시뮬레이션 성능 중요**
   - Fault Injection 시나리오 실시간 실행
   - 100ms 이하 응답시간 요구사항

3. **ISO 26262 / ASPICE 준수**
   - 요구사항 추적성 강조
   - 불필요한 신호는 검증 범위 증가로 이어짐

### 📋 운영 가이드라인

1. **신규 신호 추가 시**
   ```bash
   # 1. 레퍼런스 DBC에서 필요한 신호 확인
   # 2. vehicle_system.dbc에 복사
   # 3. alignment_review.md 업데이트
   # 4. CANoe에서 로드 테스트
   ```

2. **정기 검증 (분기별)**
   - 요구사항 변경 시 DBC 동기화 확인
   - 누락된 신호 체크
   - `alignment_review.md` 재생성

3. **레퍼런스 DBC 업데이트 대응**
   - `reference/hyundai_kia_generic.dbc` 백업 보관
   - 필요 시 선택적으로 업데이트 반영
   - 프로젝트 DBC는 독립적으로 유지

## 🚨 예외 상황

다음 경우에만 **전체 레퍼런스 사용** 고려:

1. **프로토타이핑 초기 단계**
   - 필요한 신호를 아직 확정하지 못한 경우
   - 빠른 실험이 우선인 경우

2. **전체 차량 시뮬레이션**
   - HIL (Hardware-in-the-Loop) 테스트에서 전체 ECU 연동
   - 다른 ECU 개발팀과 공유 환경

3. **레퍼런스 검증 목적**
   - Hyundai/Kia DBC 정확성 확인용

**단, 프로덕션 단계에서는 반드시 선별 추출 방식으로 전환!**
