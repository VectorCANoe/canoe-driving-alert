**올바른 답변**:

- "각 ECU는 독립적인 기능을 수행하지만, 차량 전체 서비스를 위해 데이터를 주고받아야 합니다"

### Q2: "통신이 왜 필요한가?"
**현재 우리 이해 (부족함)**:
- IVI 내부 컴포넌트 간 통신

**올바른 이해**:
- **예시 1**: 속도 정보
  - 엔진 ECU가 측정 → CAN 전송 → 클러스터/IVI가 표시
- **예시 2**: ADAS 경고
  - 카메라/레이더 ECU가 감지 → CAN 전송 → IVI가 시각적 경고
- **예시 3**: 라이팅 제어
  - IVI가 사용자 입력 수신 → CAN 전송 → BCM이 실제 조명 제어

**멘토의 비유**:
> "5명이 회의실에 모여있습니다. 입과 귀가 통신(CAN)이고, 머릿속 생각이 애플리케이션이며, 회의 결과가 차량 동작입니다"

**올바른 이해**:

- 아키텍처 다이어그램에 포함되면 안 됨

## 📐 시스템 아키텍처 레벨 구조

### Level 1: 차량 전체 시스템
```
[OTA Server] ←→ [Gateway ECU]
                      ↓
    ┌─────────────────┼─────────────────┐
    ↓                 ↓                 ↓
[Powertrain]     [Chassis]          [Body]         [Infotainment]
- Engine ECU     - ESP ECU          - BCM           - IVI
- TCU            - ABS ECU          - Door ECU      - Cluster
                 - Steering ECU     - Light ECU     - Camera ECU
```

**구성 요소**:
- Powertrain Domain: 엔진, 변속기
- Chassis Domain: 제동, 조향, 차체 제어
- Body Domain: 도어, 조명, 편의 기능
- Infotainment Domain: IVI, 클러스터, ADAS 센서
- External: OTA 서버

**연결**: CAN 네트워크 (High-Speed CAN, Low-Speed CAN)

### Level 2: 도메인별 상세
각 도메인 내 ECU 구성 및 역할

### Level 3: ECU 단위
- 각 ECU의 입출력
- 송수신 CAN 메시지 목록
- 주요 기능

### Level 4: ECU 내부 (선택적, 1-2개만)
- IVI ECU 내부 AUTOSAR 구조
- 소프트웨어 컴포넌트 상세 설계
