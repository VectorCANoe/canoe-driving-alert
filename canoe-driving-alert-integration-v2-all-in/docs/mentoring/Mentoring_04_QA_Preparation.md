## Q1: "차량 시스템이란 무엇인가?"

### 핵심 답변 (30초)
"차량은 **47개의 ECU가 CAN 네트워크로 연결된 분산 시스템**입니다.

각 ECU는 독립적인 기능을 수행하지만, 차량 전체 서비스를 위해 **CAN 메시지로 데이터를 주고받아야 합니다**.

7개 도메인으로 구성됩니다:
- **Powertrain** (7 ECU): 엔진, 변속기 제어
- **Chassis** (6 ECU): 제동, 조향, 안정성 제어
- **ADAS** (7 ECU): 카메라, 레이더 센서
- **Body** (8 ECU): 도어, 조명, 에어컨
- **Infotainment** (5 ECU): IVI, 계기판, HUD
- **Safety** (2 ECU): 에어백, 탑승자 감지
- **Gateway** (1 ECU): 네트워크 중계"

### 상세 설명 (필요시)

#### 분산 시스템의 의미
- **단일 컴퓨터가 아님**: 차량은 하나의 큰 컴퓨터가 아니라, 여러 개의 작은 컴퓨터(ECU)가 협업하는 시스템
- **독립성**: 각 ECU는 자신의 프로세서, 메모리, 소프트웨어를 가짐
- **협업**: 차량 전체 기능을 위해 데이터를 공유해야 함

#### 도메인별 역할

| 도메인 | ECU 수 | 주요 역할 | 예시 |
|--------|--------|----------|------|
| Powertrain | 7 | 동력 생성 및 전달 | EMS (엔진), TCU (변속기) |
| Chassis | 6 | 차량 제어 및 안정성 | ESP (안정성), ABS (제동) |
| ADAS | 7 | 운전 보조 | F_CAMERA (차선 유지), SCC (크루즈) |
| Body | 8 | 편의 기능 | BCM (조명), DATC (에어컨) |
| Infotainment | 5 | 정보 및 엔터테인먼트 | IVI (내비), CLU (계기판) |
| Safety | 2 | 안전 장치 | ACU (에어백), ODS (탑승자 감지) |
| Gateway | 1 | 네트워크 중계 | CGW (메시지 라우팅) |

#### 네트워크 구조
```
CAN-HS #1 (500 kbps)    CAN-HS #2 (500 kbps)    CAN-LS (125 kbps)
┌──────────────┐        ┌──────────────┐        ┌──────────────┐
│ Powertrain   │        │ ADAS         │        │ Body         │
│ Chassis      │◄──────►│ Infotainment │◄──────►│              │
│ Safety       │        │              │        │              │
└──────────────┘        └──────────────┘        └──────────────┘
       ▲                       ▲                       ▲
       └───────────────────────┼───────────────────────┘
                          Gateway (CGW)
```

## Q2: "통신이 왜 필요한가?"

### 핵심 답변 (30초)
"각 ECU가 독립적으로 동작하지만, **차량 전체 서비스를 위해서는 데이터를 공유**해야 합니다.

3가지 실제 사례로 설명드리겠습니다:
1. **속도 표시**: EMS가 측정한 속도를 CLU와 IVI가 표시
2. **ADAS 경고**: 카메라가 감지한 차선 이탈을 IVI가 시각화
3. **앰비언트 라이팅**: IVI의 사용자 입력을 BCM이 LED 제어에 반영"

### 시나리오 1: 속도 표시

#### 문제
"계기판(CLU)과 IVI가 차량 속도를 어떻게 알 수 있는가?"

#### 해결
1. **EMS (Engine ECU)**가 차량 속도 측정
2. **CAN 메시지** `EMS16 (0x260)`로 속도 전송
   - 신호: `Vehicle_Speed` (0-300 km/h, 0.01 km/h 단위)
   - 주기: 10ms (ASIL-D)
3. **Gateway**가 CAN-HS #1 → CAN-HS #2로 라우팅
4. **CLU**와 **IVI**가 수신하여 속도계 표시

#### 통신 흐름
```
EMS (CAN-HS #1) → Gateway → CLU, IVI (CAN-HS #2)
    [EMS16]         [R001]      [속도 표시]
```

#### 통신이 없다면?
- CLU와 IVI가 각각 속도 센서를 가져야 함 (비용 증가)
- 센서 값이 다를 수 있음 (일관성 문제)
- 배선 복잡도 증가

### 시나리오 2: ADAS 경고

#### 문제
"차선 이탈 시 IVI가 어떻게 경고를 표시하는가?"

#### 해결
1. **F_CAMERA (전방 카메라)**가 차선 이탈 감지
2. **CAN 메시지** `LDWS_LKAS11 (0x420)`로 이벤트 전송
   - 신호: `LDW_Status` (0: 정상, 1: 좌측 이탈, 2: 우측 이탈)
   - 주기: 100ms (ASIL-B)
3. **IVI**가 CAN-HS #2에서 수신
4. **시각적 경고** 표시 (화면 + 사운드)

#### 통신 흐름
```
F_CAMERA (CAN-HS #2) → IVI (CAN-HS #2)
    [LDWS_LKAS11]        [경고 표시]
```

#### 통신이 없다면?
- IVI가 자체 카메라를 가져야 함 (중복 투자)
- 카메라 ECU의 고급 알고리즘 활용 불가
- 시스템 복잡도 증가

### 시나리오 3: 앰비언트 라이팅 (프로젝트 특화) ⭐

#### 문제
"IVI에서 실내 조명 색상을 변경하려면?"

#### 해결 (양방향 통신)

**Forward Path (IVI → BCM)**:
1. **사용자**가 IVI에서 "SPORT" 테마 선택
2. **IVI**가 `IVI_AmbientLight (0x400)` 메시지 생성
   - Ambient_Light_R = 255 (Red)
   - Ambient_Light_G = 0
   - Ambient_Light_B = 0
   - Brightness = 80 (80%)
   - Theme_Package = 0 (SPORT)
3. **Gateway**가 CAN-HS #2 → CAN-LS로 라우팅 (Rule R011)
4. **BCM**이 수신하여 LED 드라이버 제어

**Feedback Path (BCM → IVI)**:
5. **BCM**이 `BCM_LightControl (0x520)` 메시지 생성
   - Ambient_R_Actual = 255
   - Ambient_G_Actual = 0
   - Ambient_B_Actual = 0
   - Ambient_Light_Active = 1 (ACTIVE)
6. **Gateway**가 CAN-LS → CAN-HS #2로 라우팅 (Rule R021)
7. **IVI**가 수신하여 "Ambient Light: SPORT (Red, 80%) ✓ Active" 표시

#### 통신 흐름
```
User → IVI → Gateway → BCM → LED
              (R011)

LED → BCM → Gateway → IVI → User
            (R021)
```

#### 타이밍
- **총 레이턴시**: ~10-15ms
- **사용자 체감**: 즉각 반응

#### 통신이 없다면?
- IVI가 직접 LED를 제어해야 함 (하드웨어 복잡도)
- 실제 LED 상태 피드백 불가 (사용자 경험 저하)
- BCM의 조명 통합 제어 불가

## Q3: "CANoe의 역할은?"

### 핵심 답변 (30초)
"CANoe는 **실제 차량 없이 가상 차량 환경을 구축하는 시뮬레이션 도구**입니다.

주요 역할:
1. **ECU 시뮬레이션**: 47개 ECU를 CAPL 노드로 구현
2. **메시지 송수신 검증**: CAN 통신 흐름 확인
3. **오류 테스트**: CRC 오류, 타임아웃 등 시뮬레이션
4. **성능 측정**: 버스 로드, 레이턴시 측정

**중요**: CANoe는 아키텍처 다이어그램에 포함되지 않습니다. 개발 및 테스트 도구일 뿐, 차량 시스템의 구성 요소가 아닙니다."

### 상세 설명 (필요시)

#### CANoe vs 실제 차량

| 항목 | 실제 차량 | CANoe 시뮬레이션 |
|------|----------|-----------------|
| ECU | 실제 하드웨어 | CAPL 노드 (소프트웨어) |
| CAN 네트워크 | 물리적 배선 | 가상 네트워크 |
| 센서 | 실제 센서 | 시뮬레이션 값 |
| 비용 | 수천만 원 | 소프트웨어 라이선스 |
| 테스트 | 제한적 | 무제한 (오류 주입 가능) |

#### CANoe 활용 사례

**1. 개발 단계**
- ECU 소프트웨어 개발 전 통신 구조 검증
- 메시지 정의 및 신호 매핑 확인
- 네트워크 토폴로지 시뮬레이션

**2. 테스트 단계**
- 기능 테스트: 메시지 송수신 확인
- 오류 주입: CRC 오류, AliveCounter 타임아웃
- 성능 측정: 버스 로드, 레이턴시

**3. 통합 단계**
- 실제 ECU + 가상 ECU 혼합 테스트 (HIL: Hardware-in-the-Loop)
- 전체 시스템 통합 검증

#### 우리 프로젝트에서의 CANoe 역할

**구현 계획** (2월 14-15일):
1. **CANoe 프로젝트 생성**
   - DBC 파일 임포트 (149 messages, 1345 signals)
   - 3개 네트워크 구성 (CAN-HS #1, #2, CAN-LS)

2. **주요 ECU 노드 구현**
   - **EMS**: `EMS16 (0x260)` 주기 전송 (10ms)
   - **IVI**: `IVI_AmbientLight (0x400)` 전송 (100ms)
   - **BCM**: LED 제어 + `BCM_LightControl (0x520)` 피드백
   - **Gateway**: 라우팅 규칙 (R011, R021)

3. **시뮬레이션 Panel**
   - Control Panel: 테마 선택, Brightness 슬라이더
   - Display Panel: RGB 값, 피드백 상태

4. **테스트 시나리오**
   - 속도 표시 (EMS → CLU)
   - 앰비언트 라이팅 (IVI → BCM → IVI)
   - Gateway 라우팅 검증

#### 왜 아키텍처 다이어그램에 포함되지 않는가?

**잘못된 이해** (이전):
```
┌─────────┐
│  CANoe  │ ← 시스템 구성 요소로 착각
└─────────┘
```

**올바른 이해** (현재):
```
┌─────────────────────────────────┐
│  CANoe (시뮬레이션 도구)         │
│  ┌───────────────────────────┐  │
│  │  차량 시스템 (47 ECU)      │  │ ← 시뮬레이션 대상
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

- CANoe는 **도구(Tool)**이지, **시스템(System)**이 아님
- 실제 차량에는 CANoe가 없음
- 개발/테스트 환경에서만 사용

### 비유
"CANoe는 건축가가 사용하는 **3D 시뮬레이션 소프트웨어**와 같습니다. 실제 건물을 짓기 전에 가상으로 구조를 확인하고 테스트하는 도구입니다. 건물 설계도에 '3D 소프트웨어'를 그리지 않는 것처럼, 차량 아키텍처 다이어그램에 CANoe를 그리지 않습니다."

## 추가 예상 질문

### Q4: "프로젝트의 차별점은?"

**답변**:
"일반적인 CAN 통신 프로젝트와 달리, 저희는 **실제 차량 데이터(OpenDBC)**를 기반으로 **프로덕션급 아키텍처**를 설계했습니다.

차별점:
1. **실제 차량 기반**: Hyundai/Kia OpenDBC (146 messages, 1325 signals)
2. **표준 준수**: ISO 26262, AUTOSAR, Hyundai/Mobis
3. **프로젝트 특화**: 앰비언트 라이팅 양방향 통신 ⭐
4. **4-Level 아키텍처**: Level 1 (전체) → Level 2 (도메인) → Level 3 (통신) → Level 4 (ECU 상세)
5. **Gateway 라우팅**: 50+ 규칙, 우선순위, 레이턴시 관리"

### Q5: "가장 어려웠던 점은?"

**답변**:
"초기에는 **IVI ECU 하나를 완벽하게 구현**하는 것이 목표였습니다. 하지만 멘토님 피드백을 통해 **차량 전체 시스템 이해**가 핵심임을 깨달았습니다.

어려웠던 점:
1. **프로젝트 방향 전환**: 단일 ECU → 47개 ECU 분산 시스템
2. **OpenDBC 분석**: 1325개 신호를 도메인별로 분류
3. **Gateway 라우팅**: 네트워크 간 메시지 흐름 설계
4. **표준 준수**: ISO 26262, AUTOSAR 규격 학습

극복 방법:
- 멘토 피드백을 즉시 반영
- OpenDBC 실제 차량 데이터 분석
- Hyundai/Mobis 표준 벤치마킹
- 4-Level 구조화로 복잡도 관리"

### Q6: "V-사이클 프로세스 경험은?"

**답변**:
"전체 V-사이클을 경험했습니다:

**Left Side (설계)**:
1. **요구사항**: 차량 분산 시스템 이해, CAN 통신 검증
2. **시스템 설계**: Level 1 (47 ECU, 7 도메인)
3. **상세 설계**: Level 2-3 (도메인별, 통신 구조)
4. **구현**: Level 4 (IVI ECU AUTOSAR) + CANoe

**Right Side (검증)** (예정):
5. **단위 테스트**: ECU 노드별 메시지 송수신
6. **통합 테스트**: Gateway 라우팅, 네트워크 통신
7. **시스템 테스트**: 앰비언트 라이팅 시나리오
8. **인수 테스트**: 성능 측정 (버스 로드, 레이턴시)"
