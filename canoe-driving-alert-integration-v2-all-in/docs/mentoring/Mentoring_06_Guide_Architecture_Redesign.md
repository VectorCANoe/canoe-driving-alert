# 가이드: 샘플 기반 컨셉 디자인 및 아키텍처 재설계

멘토가 제공한 `Project Result_Sample.xlsx`와 피드백을 바탕으로, 우리 프로젝트의 아키텍처를 어떻게 재구성해야 하는지 구체적인 가이드를 제시합니다.

---

## 1. 접근 방식의 변화 (Top-Down)

기존에는 "어떤 기능을 만들까? -> 아, 이런 메시지가 필요하겠네 -> DBC 만들자" 순서였다면,
**변경 **는 "전체 시스템은 무엇인가? -> 도메인별 기능은? -> 도메인 간 어떤 데이터가 오고 가는가? -> (나중에) 그게 어떤 메시지인가?" 순서여야 합니다.

---

## 2. 컨셉 디자인 (Concept Design) 작성 가이드

샘플의 `0301_SysFuncAnalysis` 시트를 기반으로 작성합니다.

### 2.1 도메인 정의 (Level 1)
먼저 우리 차량 시스템을 구성하는 큰 그룹(도메인)을 정의합니다. 샘플에서는 다음과 같이 나누었습니다:
- **Powertrain**: 엔진, 변속기 등 동력 전달
- **Chassis**: 조향(Handle), 제동(Brake), 가속(Accel) 등 주행 제어
- **Body**: 램프, 창문, 시트, 공조(HVAC) 등 차체 제어
- **Infotainment**: 클러스터(계기판), IVI(내비게이션) 등 사용자 인터페이스

### 2.2 ECU 및 기능 정의 (Level 2)
각 도메인 안에 포함될 ECU를 나열하고, 각 ECU가 **어떤 입력을 받아 어떤 출력을 내는지**를 문장으로 정의합니다.

**작성 예시:**

| 도메인 | ECU 이름 | 기능 상세 (Function Description) | 비고 |
| :--- | :--- | :--- | :--- |
| **Powertrain** | Engine Unit | - 가속 페달 값을 받아 엔진 출력을 조절<br>- 현재 RPM 및 엔진 온도를 측정하여 출력 | |
| **Body** | Door Unit | - 운전석/조수석 도어 스위치 입력을 감지<br>- 창문 모터를 구동하여 창문 개폐 제어 | |
| **Infotainment** | IVI Unit | - 사용자 터치 입력을 받아 미디어/내비게이션 제어<br>- 차량 상태 정보(RPM, 속도, 문열림)를 수신하여 화면에 표시 | |

> **핵심 포인트**: 여기서 "CAN 메시지 ID 0x123을 보낸다"라고 쓰면 **안 됩니다**. "RPM 정보를 보낸다"와 같이 **데이터의 흐름** 관점에서 서술하세요.

## 3. 아키텍처 다이어그램 (Architecture Diagram) 그리기

멘토가 가장 많이 지적한 부분입니다. "이더넷(Star Topology)이 아니라 CAN(Bus Topology)으로 그려야 한다"는 점을 명심하세요.

### 3.1 Level 1: 차량 전체 시스템 (Vehicle System)
- **중심**: 백본(Backbone) 역할을 하는 **CAN Bus**를 가로지르는 굵은 선으로 그립니다.
- **연결**: 각 도메인의 핵심 ECU들이 이 버스에 가지치기(Drop) 형태로 연결됩니다.
- **Gateway**: 서로 다른 통신 속도나 프로토콜(CAN FD <-> CAN)을 연결하는 경우 중앙에 Gateway를 배치합니다.

**[잘못된 그림 - Star Topology]**
```
      [Gateway]
      /   |   \
   [ECU] [ECU] [ECU]
```
(모든 선이 Gateway 점 하나로 모임)

**[올바른 그림 - Bus Topology]**
```
-----------------[ CAN Bus ]-----------------
    |          |          |          |
 [Engine]   [Brake]    [Body]     [IVI]
```
(고속도로처럼 긴 선에 휴게소처럼 ECU가 붙어 있음)

### 3.2 Level 2: 도메인 아키텍처 (Domain Architecture)
각 도메인 내부의 연결성을 보여줍니다.
- 예: Body 도메인 내부에 여러 ECU(Door, Lamp, Seat)가 있다면 이들이 어떻게 묶여 있는지 표현합니다.
- 우리 프로젝트 규모에서는 L1과 L2가 거의 유사할 수 있으나, **논리적 그룹핑**을 보여주는 것이 중요합니다.

## 4. 네트워크 흐름 정의 (Network Flow Definition)

샘플의 `0302_NWflowDef` 시트를 기반으로 작성합니다. 아키텍처가 그려졌다면, 그 위를 흐르는 **데이터**를 정의합니다.

**작성 예시:**

| Source (보내는 곳) | Destination (받는 곳) | 정보 (Data/Signal) | 설명 |
| :--- | :--- | :--- | :--- |
| Engine Unit | IVI Unit | Engine RPM | RPM 게이지 표시용 |
| Door Unit | IVI Unit | Door Open State | 문 열림 경고등 표시용 |
| IVI Unit | HVAC Unit | A/C On Command | 화면 터치로 에어컨 켜기 |

> **작업 순서**:
> 1. 위 표를 먼저 완성합니다.
> 2. 그 다음, 이 흐름을 구현하기 위해 어떤 메시지(Message)와 어떤 시그널(Signal)이 필요한지 구체화합니다.
> 3. 마지막으로 DBC 파일을 만듭니다. (우리는 이미 DBC가 있지만, 이 표와 맞는지 역검증해야 합니다.)
