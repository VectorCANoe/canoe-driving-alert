## 1. 왜 별도 가이드로 분리하는가

- 02/04 본문은 샘플 표준 형식과 산출물 성격을 유지해야 한다.
- 작성 원칙/금지 예시는 반복 안내가 필요하지만, 본문에 과도하게 넣으면 문서 목적이 흐려진다.
- 따라서 BP는 `tmp`에 별도 보관하고, 본문은 결과물 중심으로 유지한다.

## 2. 02 문서 작성법 (Concept Design)

### 2.1 02 문서의 역할

- 01(What)을 시스템 컨셉/구조/경계 관점의 How(상위 설계)로 전개하는 문서
- 구현 상세가 아니라 아키텍처 선택 근거와 동작 개념을 고정하는 문서

### 2.2 02 작성 원칙

1. 기능 요구를 시나리오/레이어/블록 관점으로 정리한다.
2. 경계와 인터페이스를 명확히 한다(입력원, 출력대상, 네트워크 계층).
3. 우선순위/중재/복귀 같은 시스템 규칙은 개념 수준으로 기록한다.
4. 구체 코드/스크립트/패킷 절차는 쓰지 않는다.

### 2.3 권장 02 문서 구조

1. 문서 목적/범위/제약
2. 시스템 컨셉(레이어/블록)
3. 핵심 시나리오 동작
4. 아키텍처 선택 근거(왜 이 구조인지)
5. 네트워크 경계(CAN/Ethernet)
6. 제외 범위 명시

### 2.4 02 금지 항목

- CAPL 함수/이벤트 핸들러 직접 기재
- `sysvar::...` 강제 주입 절차
- UDS 서비스 시퀀스/디버깅 절차 나열

## 3. 04 문서 작성법 (SW Implementation)

### 3.1 04 문서의 역할

- 코드 자체 설명서가 아니라, `03/0301/0302/0303/0304`를 실제 구현으로 연결하는 증거 문서
- 핵심: How를 구현 관점으로 고정하고 추적성 유지

### 3.2 04 작성 원칙

1. 구현 코드를 그대로 복붙하지 않는다.
2. 메시지/변수/함수의 계약(입력/출력/주기/예외)만 문서화한다.
3. 범위 외 항목(OTA/UDS/DoIP)은 완전 제외한다.
4. 추적 체인은 반드시 유지한다: `Req -> Func -> Flow -> Comm -> Var -> Code -> UT/IT/ST`.

### 3.3 권장 04 문서 구조

1. 문서 목적/범위/제약
2. 구현 아키텍처 요약(Option1: `ETH_SWITCH + GW + CAN`)
3. 노드별 구현 책임 표(공식 상단표 스타일)
4. 구현 매핑 표(하단 상세)
5. 실행/타이밍 표
6. 상태/중재 처리 표
7. 예외/장애 처리 표
8. 구현 산출물 인덱스

### 3.4 04 핵심 표 템플릿

- 구현 매핑 표:
  - `Func ID | Module(File) | Input Var | Output Var | Flow/Comm | Code Ref | Test Ref`
- 실행/타이밍 표:
  - `Task/Timer | Period | Trigger | Timeout | Fail-safe`
- 상태/중재 처리 표:
  - `상태 | 진입 조건 | 출력 | 해제 조건 | 관련 Func/Req`
- 예외/장애 처리 표:
  - `고장 조건 | 감지 | 강등 동작 | 로그 | 검증 케이스`

### 3.5 04 금지 항목 (재발 방지)

- `sysvar::...=1` 같은 강제 주입 절차
- UDS 서비스 시퀀스 나열
- 디버깅용 임시 로직/명령문

---

## 4. 공통 추적성 규칙

- 필수 체인: `Req -> Func -> Flow -> Comm -> Var -> Code -> UT/IT/ST`
- 랜덤 샘플 감사 시 체인 단절이 없어야 한다.
- 상단 공식표는 샘플 형식을 유지하고, 상세 추적은 하단 표로 분리한다.
