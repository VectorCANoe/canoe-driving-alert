# Mentoring 08 Action Plan (Confirmed)

## 0. 결론 요약
- `01/03 전부 먼저 완료`도 비효율, `DBC만 먼저`도 비효율이다.
- **최적 순서는 하이브리드**:
  - `01 최소 고정 -> 03/0301 최소 고정 -> 도메인별 DBC + ETH 인터페이스 명세 -> 0302/0303 -> 0304 -> 05~07`
- 핵심은 번호 1:1 집착이 아니라 **맥락 추적 + 구현 완성도**다.

## 1. 멘토 피드백 반영 포인트
1. 차량 기능이 너무 적어서 포트폴리오 설득력이 약함  
2. DBC는 네트워크(도메인) 단위로 분리해야 함  
3. Ethernet은 DBC가 아니라 별도 인터페이스 명세로 관리해야 함  
4. ECU를 RX/TX로 쪼개지 말고 단일 ECU로 정의해야 함  
5. UT/IT/ST는 완벽 분할보다 프로젝트 완성(구현) 우선으로 단순화 가능

## 2. 확정 실행 순서

### Step 1) 01 최소 보강 (What)
- 차량 기본 기능 요구를 추가한다. (핵심 주제 보조용)
- 예: 엔진/브레이크/기어/조향/창문/비상등/클러스터
- 주제(주행상황 경고)보다 과도하게 확장하지 않는다.

### Step 2) 03/0301 최소 보강 (How)
- 확장된 ECU와 기능 책임을 반영한다.
- RX/TX 분리 네이밍을 제거하고 ECU 단일 객체로 정리한다.

### Step 3) 통신 원본 확정
- CAN: 도메인(네트워크)별 DBC 작성
- Ethernet: 별도 인터페이스 명세 작성 (E100/E200, 0x510/0x511/0x512)

### Step 4) 0302/0303 작성
- CAN 영역: 도메인별 DBC에서 역추출
- Ethernet 영역: ETH 인터페이스 명세 기준 작성
- 누락 없이 Flow/Comm 정합 확인

### Step 5) 0304 정합
- 0302/0303 기준으로 Var 체인 정리
- `Req -> Func -> Flow -> Comm -> Var` 단절 여부 점검

### Step 6) 테스트 문서 마무리 (05~07)
- 시간 우선순위:
  - 우선 `UT + ST`
  - `IT`는 최소 핵심 체인만 유지하거나 ST와 통합
- 목표는 번호 맞추기가 아니라 커버리지/설명 가능성 확보

## 3. 문서/구현 경계 원칙
- `01`: What만 작성
- `03+`: How만 작성
- DBC: CAN 계약만 작성
- Ethernet 계약: 인터페이스 명세로 분리
- PPT: 제목과 그림 역할 일치 (오버뷰 vs 통신구조)

## 4. 체크포인트
- C1: 01에 차량 기본 기능 요구 추가 완료
- C2: 03/0301에 ECU 확장 반영 완료
- C3: 도메인별 CAN DBC 초안 완료
- C4: ETH 인터페이스 명세 초안 완료
- C5: 0302/0303 작성 완료 (CAN=DBC, ETH=ETH spec)
- C6: 0304/05~07까지 체인 단절 없음

## 5. Done Definition
- 문서: 00~07이 큰 흐름에서 일관되고 설명 가능
- 통신: CAN 계약은 DBC와 1:1 정합
- 경계: Ethernet 계약은 ETH 명세와 1:1 정합
- 테스트: 최소 UT/ST로 시나리오 시연 가능
