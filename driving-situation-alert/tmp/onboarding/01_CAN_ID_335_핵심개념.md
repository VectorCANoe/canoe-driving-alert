# 3/3/5 핵심개념 (쉬운 버전)

## 한 줄 요약
- ID 배정은 설계 규칙
- 버스중재는 CAN 물리/프로토콜 규칙
- 기능중재는 SW 로직 규칙

---

## 1. ID 배정 규칙 (설계 시점)

목적:
- CAN ID를 어떤 기준으로 부여할지 결정

순서:
1. Owner/도메인 경계
2. 안전/검증 영향
3. Tier/Group/Index(3/3/5) 인코딩

핵심:
- 여기서는 "ID를 어떻게 설계할지"를 정한다.

---

## 2. 버스중재 규칙 (CAN 실행 시점)

목적:
- 동시에 송신될 때 누가 먼저 전송되는지 결정

기준:
- CAN ID 값만 사용
- payload 값은 버스중재 순서에 영향 없음

핵심:
- 여기서는 "버스에서 실제 전송 순서"가 정해진다.

---

## 3. 기능중재 규칙 (SW 실행 시점)

목적:
- 경고/감속보조/Fail-safe 같은 기능 판단

기준:
- payload
- 타이머
- 상태 로직

제한:
- 기능중재는 버스중재 순서를 바꿀 수 없다.

---

## 4. 3/3/5 비트 의미

- 앞 3비트: Tier (우선순위 클래스)
- 중간 3비트: Group (기능/도메인 분류)
- 뒤 5비트: Index (해당 Tier/Group 내부 슬롯)

계산식:
- `ID = (Tier << 8) | (Group << 5) | Index`

Index 범위:
- `0~31` (5bit = 32개 슬롯)
- `0~34`가 아님

---

## 5. 가장 많이 헷갈리는 포인트

Q1. "Tier는 설계, Group은 CAN 실행, Index는 SW 실행" 맞나요?
- 아니요.
- Tier/Group/Index는 모두 설계 인코딩이다.
- CAN 실행 중재는 최종 ID 전체 숫자로만 동작한다.
- SW 기능중재는 payload/타이머/상태 로직으로 동작한다.

Q2. "긴급이면 Group 1로 옮겨야 하나요?"
- 아니다.
- 긴급 우선의 기본 판정 축은 기능중재(WARN_ARB_MGR)다.
- 버스 우선순위 상향은 별도 요구/게이트 근거가 있을 때만 한다.

Q3. "Diag가 이름에 있으면 Group 7 고정인가요?"
- 아니다.
- Group 결정은 Owner/도메인 경계와 안전 경로를 우선한다.
