# 03.03 Mentoring 정리 (MET_40_2026.03.03)

기준 회의록: `docs/meeting-notes/MET_40_2026.03.03.txt`  
기준: 멘토(`참석자 1`) 발화만 추출

## 1) 멘토 핵심 지시사항

1. 중간보고 제출물은 `PPT + 엑셀 + DBC` 기준으로 준비.
2. DBC는 도메인 분리 제출이 기본.
3. 엑셀 양식은 열/내용 수정 가능(비교 가능성만 유지).
4. Ethernet은 CAN DBC로 처리하지 말고, 커뮤니케이션 매트릭스/인터페이스 계약으로 관리.
5. Ethernet 상세 프로토콜(ARXML/상위 절차) 깊이 확장은 이번 범위에서 과도할 수 있음.
6. `test_can`류 명칭/구조는 오해 없게 정리(개발 편의 분리는 가능하나 제출 해석 명확화 필요).
7. 네트워크 도식은 Ethernet 버스선 표현 금지, Connectivity Unit 기준 포트 연결(스타형)로 표현.
8. Panel은 꼭 유지: Unity 병행 가능하지만 Panel 등한시 금지.
9. Panel 우선순위: `차량 화면 -> 제어 패널 -> 상태 모니터`.
10. 인사이트/KPI 분석보다 V-Model 체인 폐쇄(요구-기능-구현-테스트-확인)가 핵심.
11. `Req_041~043`, `Req_112`는 기능요구보다 검증환경/품질요구 성격으로 분리 인식.
12. 네트워크/커뮤니케이션 문서에 주기(Period)와 전송이벤트(Event Trigger) 명시.
13. “우선순위 중재” 용어는 CAN arbitration 혼동 가능 -> 서비스 레벨 표현으로 분리.
14. 요구사항 리미트는 없으나 과세분화는 통폐합 가능.
15. CAN ID 설계는 고정 정답 없음. 팀 룰과 설계 근거를 문서에 명시.
16. Ethernet 라이선스는 별도 발급 지원 요청 진행.
