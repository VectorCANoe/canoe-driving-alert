# 문서 작성 가이드라인 보고서 (문제점/개선방향/추적성)

**프로젝트**: Driving Situation Linked Real-Time Warning System  
**작성일**: 2026-02-26  
**범위**: `driving-situation-alert` 문서 체계 전체(특히 03~07)

---

## 1. 왜 문제가 되었는가

이전 문서 일부는 "프로젝트 산출물"이 아니라 "개발자 구현 메모" 수준으로 작성되어 있었다.

대표 문제 유형:
- 코드/스크립트 레벨 직접 기술: `sysvar::... = 1`, 강제 주입 절차
- 프로토콜 절차 나열 과다: `DoIP -> UDS 0x10 -> 0x27 -> 0x34 -> 0x36 -> 0x37 -> CRC -> Rollback`
- 비트/주기/임계치 로직을 설계 의도 없이 구현 상세처럼 단정 서술
- 현재 고정 스코프(CANoe SIL, CAN+Ethernet, 옵션1 아키텍처)와 불일치 항목 잔존

핵심 문제는 "문서 레벨 혼합"이다.
- 요구사항/설계/구현/검증 문서의 경계가 무너지면
- 추적성 감사(랜덤 Req 샘플)에서 체인이 끊기고
- 변경 시 영향 분석이 불가능해진다.

---

## 2. 문서의 핵심 가치(무조건 유지)

1. 레벨 분리: `01=What`, `03+=How`, `04=Code 구조`, `05~07=검증 증거`
2. 표준 양식 준수: 상단 공식표는 샘플 형식 그대로 유지
3. 추적성 완전성: `Req -> Func -> Flow -> Comm -> Var -> Code -> UT/IT/ST`
4. 검증 가능성: 문장/표는 반드시 테스트 가능한 조건으로 작성
5. 범위 일치성: CANoe SIL, CAN+Ethernet, 옵션1 아키텍처 고정

## 4. 추적성 체인 유지 방법 (실무 규칙)

| 단계 | 식별자 | 질문 | 합격 기준 |
|---|---|---|---|
| 요구 | Req ID | 무엇을 만족해야 하는가 | 요구가 단일 의미로 검증 가능 |
| 기능 | Func ID | 누가 수행하는가 | Req와 1개 이상 연결 |
| 흐름 | Flow ID | 데이터가 어디로 가는가 | Func와 송수신 경로 연결 |
| 통신 | Comm ID | 어떤 메시지/신호인가 | ID/DLC/Bit/Range 일관 |
| 변수 | Var ID | 내부 상태로 어떻게 관리되는가 | Comm/Flow/Func/Req 역추적 가능 |
| 구현 | Code Ref | 어디에 구현되었는가 | Var/Comm와 참조 연결 |
| 검증 | UT/IT/ST ID | 어떻게 증명하는가 | 실패 시 역추적 가능 |

랜덤 감사 규칙:
- 임의 `Req ID`를 골라 `ST -> IT -> UT -> Code -> Var -> Comm -> Flow -> Func -> Req` 왕복이 끊기지 않아야 한다.

## 5. 표 작성 원칙 (중요)

- 상단: 공식 표준 양식 100% 유지(샘플과 동일 열 구성)
- 하단: 상세/운영/추적 보강표 별도 작성
- 표 간 역할 분리:
  - 상단은 "공식 산출물 가독성"
  - 하단은 "개발/검증 추적성"

0304 권장 패턴:
- 상단: `Namespace=도메인`, `Name=순수 변수명`
- 하단: `Internal Name`, `Bus Path`, `Unit/Scale/Endian`, `Var-Comm-Flow-Func-Req`
