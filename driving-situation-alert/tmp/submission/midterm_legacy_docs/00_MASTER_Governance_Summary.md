> 본 문서는 중간 제출 심사 관점에서 "이 프로젝트가 왜 신뢰 가능한가"를 설명하기 위한 운영 원칙 요약본입니다.  
> 상세 정책/전수표는 SoT 원문(`00c/00d/00e/00f/00g`)을 기준으로 관리합니다.

## 한눈에 보기

| 질문 | 답변 |
|---|---|
| 이 문서는 왜 필요한가? | 프로젝트가 어떤 기준으로 문서를 운영하고 검증하는지 한 번에 설명하기 위해 필요하다. |
| 무엇을 보장하는가? | 용어/ID/상태라벨/추적 체인이 문서 간에 일관되게 유지됨을 보장한다. |
| 어디까지가 제출 범위인가? | 본문은 요약 원칙만 담고, 상세 전수 근거는 SoT 원문(00c~00g)으로 분리한다. |

## 중간보고 핵심

| 구분 | 핵심 메시지 |
|---|---|
| 운영 원칙 | 기능 확장보다 문서 간 정합성과 추적 일관성을 우선한다. |
| 정책 범위 | ECU 명명, CAN ID, RTE 명명, 상태 라벨 규칙을 동일 기준으로 유지한다. |
| 심사 대응 | 제출본은 요약, 상세 근거는 SoT 원문으로 즉시 확장 가능한 구조를 유지한다. |

## 1) 운영 철학

- 우리는 기능 추가보다 "일관성 있는 추적 체인"을 우선한다.
- 한 문서만 맞고 다른 문서가 어긋나는 상태를 허용하지 않는다.
- 검증 가능한 문장만 남기고, 모호한 선언형 문장을 줄인다.

## 2) 품질·안전 적용 수준

- 본 프로젝트는 CANoe SIL 기반 학습/시연 프로젝트이며 인증 취득 주장 문서는 아니다.
- 그러나 실무 리뷰를 대비해 ISO 26262/ASPICE의 문서 체계와 검증 사고방식을 반영한다.
- 핵심 목표는 "심사자가 근거를 끝까지 따라갈 수 있는 구조"를 만드는 것이다.

## 3) 문서 체인 거버넌스

고정 체인:

`Req -> Func -> Flow -> Comm -> Var -> Code -> UT -> IT -> ST`

운영 원칙:

- `01`은 What, `03+`는 How로 분리한다.
- 체인 변경 시 단일 문서만 수정하지 않고 연관 문서를 동시 반영한다.
- 제출본은 대표 매핑 중심으로 간결화하고, 전수 근거는 원문 SoT에서 유지한다.

## 4) 상태 관리 원칙 (Active / Legacy / Pre-Activation)

| 상태 | 의미 | 제출 시 처리 |
|---|---|---|
| Active | 현재 구현/검증 대상 | 본문에 유지 |
| Legacy | 통합/분할로 대체된 이전 항목 | 삭제하지 않고 매핑 근거 유지 |
| Pre-Activation | 설계 선반영, 실행 전 상태 | 범위를 명시하고 Planned/Ready로 관리 |

## 5) 핵심 기술 정책 요약

### ECU 명명 정책

- Canonical 명칭만 사용한다.
- 형식은 `UPPER_SNAKE_CASE`와 역할 접미사(`_GW`, `_CTRL`, `_MGR`, `_TX`, `_RX`)를 따른다.
- Validation 노드는 `VAL_*` 접두를 사용한다.

### CAN ID 정책

- 도메인 분리와 3/3/5 배정 원칙을 함께 적용한다.
- CAN ID와 Ethernet 논리 ID는 목적이 다르므로 분리 해석한다.
- ID 충돌/중복을 허용하지 않는다.

### RTE 명명 정책

- Canonical -> AUTOSAR shortName -> RTE 생성명 3계층을 유지한다.
- 임의 축약으로 의미가 변형되는 것을 금지한다.

## 6) HARA와 테스트 연결 원칙

- 위험 이벤트는 HARA에서 정의하고, 요구/설계/시험 문서에서 동일 ID로 추적한다.
- 핵심 위험 시나리오는 "반응시간 + 상태전이 + 복귀조건"까지 검증 기준으로 관리한다.
- 테스트 결과 해석은 단순 Pass/Fail보다 "근거 경로가 닫혔는지"를 우선 확인한다.

## 7) 심사자 관점에서의 신뢰 포인트

1. 요구부터 테스트까지 ID 기반으로 역추적이 가능하다.
2. 용어(ECU명, ID, 상태 라벨)가 문서마다 다르게 쓰이지 않는다.
3. 실행 전 항목(Pre-Activation)과 실행 완료 항목(Active)을 혼동하지 않게 분리했다.
4. 제출본은 간결하지만, 필요 시 원문 SoT로 즉시 확장 증빙이 가능하다.

## 8) 제출 운영 방식

- 중간 제출에는 본 문서를 00 정책 요약으로 사용한다.
- 상세 근거 요청 시 `governance` 폴더의 00c/00d/00e/00f/00g 문서를 부록으로 제시한다.
