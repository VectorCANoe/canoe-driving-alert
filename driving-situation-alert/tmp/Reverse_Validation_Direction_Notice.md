# 역방향 검증 진행 방향 (합의안)

## 1. 핵심 결정
- 문서 체인 완성(`01 -> 03/0301/0302/0303/0304 -> 04 -> 05/06/07`)을 최우선으로 한다.
- 코드 업그레이드(ETH UdpSocket 실구현)는 선택 과제로 분리한다.

## 2. 작업 구분
- [A] 문서 체인: 멘토 검토 대상, 필수
- [B] 코드 고도화: 가점 요소, 선택

## 3. 즉시 실행 순서
1. `FZ_001~FZ_012` 사전 가능범위 점검 수행
2. 결과 기반으로 `05_Unit_Test.md` 확정
3. `06_Integration_Test.md` 확정
4. `07_System_Test.md` 확정
5. 최종 체인 폐쇄 검증: `Req -> Func -> Flow -> Comm -> Var -> Code -> UT/IT/ST`

## 4. 기술 경로 원칙
- 공식 본선 통신 계약은 `0302/0303/0304` 기준을 유지한다.
- SIL 단계에서는 Panel/sysvar 입력을 허용한다.
- 목표 경로는 ETH `UdpSocket` 입력으로 점진 전환한다.
- `0x130 Proxy`는 스파이크/임시 검증 전용으로만 사용한다.

## 5. Go/No-Go 기준
- Go: `FZ_001~FZ_012` 모두 Pass 또는 필수 항목 Pass + 보완계획 명시
- No-Go: 필수 항목 Fail 시 04/0302~0304 추적 갱신 후 재시험
