## 작성 원칙

- 본 문서는 고객 관점의 요구(What)만 정의한다.
- 구현 방법(함수/코드/프로토콜 상세)은 기능명세(03 문서)에서 정의한다.
- 상단 통합 기본요구사항은 `Req_001~Req_040`을 디폴트 기준으로 사용한다.
- 차량 기본 기능 확장요구는 `Req_101~Req_107`, `Req_109~Req_111`, `Req_113~Req_119`를 디폴트 기준으로 사용한다.
- Part별 요구사항은 통합 기본요구사항을 재분류한 구조다.
- `Req_041~Req_043`, `Req_112`는 기능요구 본문이 아닌 하단 `Part 6. 검증/인수 조건`에서 관리한다.
- ASPICE SYS.2.BP5 기준으로 각 Req에 대해 VC(Verification Criteria)를 정의하고, 05/06/07에서 VC ID로 검증 증적을 연결한다.
- 본 사이클의 활성 요구사항 범위는 `Req_001~043`, `Req_101~107`, `Req_109~124`로 유지한다.
- 요구 분류/안전 프로파일 기준은 `00c_Req_Classification_and_Safety_Profile.md`, HARA 근거는 `00d_HARA_Worksheet.md`를 따른다.

## 요구사항 사용 가이드

- 대외 설명/발표 기준 문서: 상단 `통합 기본요구사항(Req_001~Req_040)`
- 차량 기본 기능 확장 기준: 상단 `차량 기본 기능 요구사항(Req_101~Req_107, Req_109~Req_111, Req_113~Req_119)`
- 검증/인수 조건 기준: 하단 `Part 6. 검증/인수 조건(Req_041~Req_043, Req_112)`
- 내부 작업 분배 기준: 하단 `Part별 요구사항`
- 상세 설계 연결 시: 각 Req ID를 03/0301/0302/0303/0304 및 05/06/07에 1:1 매핑
- 팀 운영 규칙 1: 물리 하드웨어 없이 CANoe 가상환경에서 검증한다.
- 팀 운영 규칙 2: 검증 네트워크 범위는 CAN + Ethernet으로 고정한다.
- 팀 운영 규칙 3: 주요 시나리오별 합격/불합격 기준을 사전에 정의한다.
- 팀 운영 규칙 4: 요구사항별 검증 결과를 추적 가능한 형태로 기록한다.
- 팀 운영 규칙 5: 임의 Req를 점검해도 문서 추적이 끊기지 않게 유지한다.
- 팀 운영 규칙 6: 프로젝트 종료 시 충족 여부를 단일 요약 문서로 확인 가능하게 정리한다.
