# 중간감사 멘토 설명용 요약 (한국어)

## 1. 문서 분석 결과 (TMP_MID_AUDIT_MAIN 기준)
- 중간감사 기준선은 `03/0301/0302/0303/0304` 5개 문서를 고정 집합으로 관리하는 구조입니다.
- 핵심 운영 원칙은 단일 수정이 아니라 **체인 단위 수정**입니다.  
  `Req -> Func -> Flow -> Comm -> Var -> Code -> UT/IT/ST`
- 랜덤 Req 질의 대응 방식(샘플 5개, 체인 역추적 순서)까지 이미 포함되어 있어 감사 대응 형태가 맞습니다.

## 2. 멘토께 설명할 핵심 포인트
### A. 아키텍처 의도
- 판단 로직은 `ADAS_WARN_CTRL`, `WARN_ARB_MGR`, `NAV_CONTEXT_MGR`, `EMS_ALERT`에 집중.
- 출력/표시는 `BODY_GW`, `IVI_GW`, Panel 쪽에서 수행.
- 즉, **로직과 렌더링 분리**를 유지했습니다.

### B. EMS 표기 원칙
- 상위 문서 표기는 논리 단말 `EMS_ALERT`로 통일.
- 구현 상세는 하단 매핑에서 `EMS_POLICE_TX`, `EMS_AMB_TX`, `EMS_ALERT_RX`로 분해.
- 문서 가독성과 구현 추적성을 동시에 확보한 방식입니다.

### C. 네트워크/통신 현실 반영
- CANoe.CAN 라이선스 제약으로 Ethernet 일부는 `eth_backbone_can_stub.dbc`로 운반.
- 하지만 상위 서비스 계약 SoT는 `10_ETHERNET_BACKBONE_SSoT.md`로 유지.
- 최근 동기화로 `0x064/0x11F/0x232`는 문서(0302/0303/0304)와 DBC를 맞췄습니다.

## 3. 멘토 질의 대응 스크립트 (권장 순서)
1. 해당 Req를 먼저 확인 (`01`)
2. 대응 Func/노드 확인 (`03`, `0301`)
3. Flow/Comm 확인 (`0302`, `0303`)
4. 변수 확인 (`0304`)
5. 구현 파일/테스트 증적 확인 (`04`, `05/06/07`)

## 4. 짧은 발표 멘트 (30초)
> 이번 중간감사는 03계열 5개 문서를 기준선으로 고정하고, 변경은 반드시 Req-Func-Flow-Comm-Var-Test 체인 단위로 관리했습니다.  
> 상위 문서는 EMS를 논리 단말로 통일하고, 구현 상세는 TX/RX 모듈로 분리해 감사 가독성과 코드 추적성을 동시에 확보했습니다.  
> 또한 CANoe 라이선스 제약 구간은 CAN-stub로 운반하되, Ethernet 서비스 계약 SoT를 유지해 문서-DBC-구현 정합을 맞췄습니다.

## 5. 회의 직전 체크

- 랜덤 Req 2~3개만 추가 리허설 (설명 순서 고정)
