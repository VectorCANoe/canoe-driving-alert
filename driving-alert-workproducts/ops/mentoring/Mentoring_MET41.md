  - 멘토: "도메인 헬스"와 "경로 상태"라는 표현이 모호함. 실제 통신에서는 end node가 경로 상태를 알 수 없음.
  - 팀원: Fail-safe 관점에서 구현했으며, 워딩을 "통신 상태 확인"으로 수정 예정.
- **요구사항 통폐합 수준:**
  - 멘토: 너무 세분화하면 관리가 어려우므로 적정 수준에서 통합하는 것이 좋음.
  - 팀원: 추적성 유지를 위해 레거시 표기 후 추후 정리 예정.

### 2.2. ID 할당 규칙 (3-3-5 포맷)
- **설계 내용:**
  - 상위 3비트: 우선순위 대역 (실시간 제어 > 경고 시스템 > 파워트레인/확장)
  - 중간 3비트: 기능 분류 (게이트웨이, 운전자 명령, 차량 동역학, 바디 편의, IVI 등)
  - 하위 5비트: 개별 메시지 번호
- **멘토 피드백:**
  - 11비트 표준 포맷으로 102개 메시지는 충분하나, 확장 시 29비트 확장 포맷 고려 가능.
  - 굳이 지금 바꿀 필요 없으며, 필요시 디펜스 포인트로 "확장 포맷 사용 가능" 정도만 기억.

### 2.3. Ethernet 라이선스 현황
- **상태:** 독일 본사 승인 지연으로 아직 미발급
- **예상:** 다음 주 중 발급 예정
- **대응:** 현재 CAN으로 대체 구현, Ethernet 전환 시 리팩토링 계획 수립

### 2.4. 검증(Verification) 프로세스
- **검증의 목적:**
  - 멘토: "요구사항대로 동작하는가?"만 확인하면 됨. Pass/Fail 외에 복잡한 분석 불필요.
  - 복잡한 로그 분석이나 대조군 비교는 실제 차량이 아니면 의미 없음.
- **테스트 방법:**
  - CAPL로 구현한 테스트 기능으로 충분
  - CANoe의 Test Feature Module도 사용 가능하나, 버전 호환성 확인 필요
  - CICD 도입 시 테스트 자동화는 가능하나, 시간 대비 효과 고려

### 2.5. CICD 도입 검토
- **팀 조사 결과:**
  - Jenkins 활용, DBC 변경 시 빌드 및 시뮬레이션 자동화 구상
  - CLI를 통한 시뮬레이션 실행 및 리포트 생성 목표
- **멘토 의견:**
  - CICD는 웹 환경에서 시작된 개념으로, 차량용 툴체인과 완벽한 통합은 어려움.
  - 경험 차원에서 도입은 좋으나, 시간 투자 대비 효과 고려 필요.
  - 최소한 테스트 자동화 수준(CI)까지만 구현하고, CD는 생략 권장.
  - 리포트 생성은 필수 (Pass/Fail 결과).

### 2.6. 패널(Panel) 개발 방향
- **현재 고민:** 정적 이미지 vs 동적 애니메이션, CANoe 패널 vs Python 연동
- **멘토 조언:**
  - 최종 평가는 **시연 영상**이 가장 중요 (1순위)
  - 패널은 시각적 이해를 돕는 도구일 뿐, 지나친 디자인에 시간 투자 불필요
  - CANoe 기본 패널로 충분, 부족하면 이미지 편집 활용
  - Python 연동은 자유도는 높지만, 안정성과 시간 고려

### 2.7. 멀티버스 구성 및 테스터 노드
- **질문:** 여러 버스에 동일한 노드(예: 테스터)를 연결해도 되는가?
- **답변:**
  - 일반 ECU는 불가능 (게이트웨이 필요)
  - 테스터 노드는 여러 버스에 연결 가능하나, 역할이 달라지므로 분리 권장
  - 불가능한 것은 아니나, 면접 시 질문 유발 가능
