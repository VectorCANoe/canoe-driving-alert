# 현업 방향성 및 의문점 답변 (Industry Direction Q&A)

**작성일**: 2026-02-20
**대상**: 팀원 토론에서 나온 의문점에 대한 대기업/표준 현업 기준 답변

> 이 문서는 팀원 토론에서 제기된 의문점을 **현업 자동차 SW 개발 표준(ISO 26262 / ASPICE / OEM 실무)** 기준으로 답변합니다.

---

## Q1. HARA와 ISO 26262를 꼭 포함해야 하나요?

> "안전쪽과 연결짓는 것이 필요한지도 궁금하네요." — 택천

### 현업 답변: Yes, BCM/OTA 수준에서는 필수

현업 자동차 SW 개발에서 **HARA(Hazard Analysis and Risk Assessment)**는 선택이 아닙니다.

- **ISO 26262:2018 Part 3**: 모든 차량 시스템 개발 시 HARA 수행이 **법적/표준 강제 사항**
- BCM의 Window Motor 과전류 미감지 → 모터 과열 → 화재 가능성 = **실제 안전 위험(Hazard)**
- OTA 실패로 BCM Bricking → Body 기능 상실 = ASIL-B 수준 안전 목표 필요

**현업 흐름**:
```
프로젝트 착수
  → HARA 수행 (ASIL 분류 → Safety Goal 도출)
  → 시스템 요구사항에 ASIL 반영
  → V-Model 설계/구현/검증 전 계층에 추적성 유지
  → Safety Case 완성 → 기능 안전 감사(Audit)
```

**단, 본 프로젝트에서의 현실적 접근**:
- ASIL-B (최고)로 제한 → ASIL Decomposition 불필요 → 복잡도 최소화
- HARA 3개(H-01~H-03), Safety Goal 2개로 간결하게 유지
- "ISO 26262를 따르는 프로세스를 경험했다"가 핵심이지, 전체 인증 취득이 목표가 아님

## Q2. 대분류 "편의 및 UX 시스템"과 저희 주제가 맞나요?

> "OTA가 ECU 소프트웨어 원격 업데이트해주는건데... 시스템 인프라에 가까운거 같아서요." — 현준

### 현업 답변: UX는 "사용자 경험 전체 계층"을 포함합니다

자동차에서 UX는 화면/UI에 국한되지 않습니다.

**현업 정의 (OEM 관점)**:
| 계층 | 예시 | UX 관련성 |
|------|------|----------|
| 표면 UX | 클러스터 경고등, 터치스크린 | 직접적 |
| 중간 계층 | 고장 감지 → 경고 표시까지의 응답 속도 | 간접적 (FTTI ≤50ms) |
| 인프라 계층 | OTA 업데이트, 진단(UDS) | **서비스 연속성 UX** |

- OTA = "차량을 방문 정비 없이 최신 상태로 유지" → **편의성의 핵심 인프라**
- 경고등 활성화 지연(FTTI) = 운전자 안전 UX
- DTC 클리어 후 경고등 소등 = 운전자 상태 피드백 UX

**발표 시 프레이밍 전략**:
> "저희 프로젝트는 사용자가 체감하는 편의 기능(경고등, OTA 원격 업데이트)이 **차량 내부 통신(LIN/CAN/Ethernet)에서 어떻게 만들어지는지** 검증합니다."

이렇게 연결하면 대분류 적합성 확보 가능.

## Q3. 가시적 결과물이 없다는 멘토 말씀, 어떻게 보완하나요?

> "주제 특성 상 가시적으로 나올 게 없어 보인다던 멘토님 말씀" — 라엘

### 현업 답변: 자동차 SW 검증의 "결과물"은 파형·로그·테스트 리포트입니다

현업에서 ECU 검증 결과물:
1. **CANoe 시뮬레이션 화면** — DTC 발생 순간 CAN 트레이스 캡처
2. **테스트 리포트 (Pass/Fail 판정표)** — In_Test_01~15, Scene.1~18 자동화 결과
3. **타이밍 파형** — LIN 10ms 주기, CAN 라우팅 ≤5ms, 경고등 ≤50ms
4. **OTA 진행률 로그** — 블록 전송 횟수, CRC 검증 결과, Rollback 확인

**비전공자 심사위원을 위한 시각화 전략**:
```
시나리오 데모 흐름:
  "지금 윈도우 모터에 과전류를 주입합니다" (TrackBar 55A 설정)
       ↓
  "CANoe에서 0.01초 이내에 DTC가 생성됩니다" (트레이스 화면)
       ↓
  "클러스터 경고등이 50ms 내에 켜집니다" (Panel Indicator)
       ↓
  "UDS로 DTC를 조회하고 클리어합니다" (Tester 화면)
       ↓
  "OTA로 펌웨어를 업데이트합니다" (진행률 표시)
```
→ 이것 자체가 매우 강력한 **실시간 데모**

**추가 권장**: 짧은 시연 영상(1~2분) 사전 제작 → 발표 중 재생

## Q4. 고장 진단 검증을 V-cycle에서 어떻게 해야 하나요?

> "고장 진단으로 가버리면 고장 진단에 관한 검증을 어떤식으로 해야할지가 감이 안잡히네요." — 현준

### 현업 답변: 고장 진단 검증 = "Fault Injection + Expected Response 확인"

현업 ECU 검증 방법론:

```
[검증 방법 3가지]

1. Fault Injection 테스트
   LIN::motorCurrent = 55A 주입
   → BCM DTC B1234 생성 확인
   → BCM_FaultStatus(0x500) 전송 확인
   Pass/Fail: DTC 생성 여부 + 50ms 이내 경고등 점등

2. Communication Fault 테스트
   LIN Slave 응답 중단 시뮬레이션 (50ms 이상)
   → DTC U0100 생성 확인
   → Fail-Safe 동작 확인
   Pass/Fail: 타임아웃 감지 + DTC 코드 일치

3. E2E 정상 복구 테스트
   DTC 클리어(UDS 0x14) 후
   → 경고등 소등 확인
   → OTA 완료 후 재시작 정상 동작 확인
   Pass/Fail: 전 과정 2회 연속 성공
```

**현업 표준 참조**:
- ISO 26262 Part 6 — SW 단위 테스트/통합 테스트 항목
- ASPICE SWE.4/5 — 소프트웨어 단위 검증 및 통합 테스트
- 본 프로젝트 `06_Integration_Test.md`의 In_Test_01~15가 바로 이 구조

## Q5. 현재 우리가 가는 방향이 대기업 실무와 얼마나 유사한가요?

### 현업 대기업(현대/기아 OEM, 현대모비스, 보쉬, 콘티넨탈) 실제 개발 흐름

```
[대기업 차량 SW 개발 프로세스 — 축약]

1단계: 시스템 요구사항 (SYS.2)
   ├── 고객 요구 → 시스템 기능 요구사항
   └── HARA → ASIL 분류 → Safety Requirements

2단계: 아키텍처 설계 (SYS.3)
   ├── ECU 할당, 네트워크 토폴로지 (CAN/LIN/Ethernet)
   └── DBC 정의, 인터페이스 명세

3단계: SW 요구사항 + 설계 (SWE.1/2/3)
   ├── 기능 분해 → 모듈 설계
   └── CAPL/C 코드 구현

4단계: 검증 (SWE.4/5, SYS.4/5)
   ├── Unit Test (CANoe CAPL Test)
   ├── Integration Test (Fault Injection, E2E 시나리오)
   └── System Qualification Test (V-Model 우측)

5단계: 도구
   ├── Vector CANoe (통신 분석/테스트)
   ├── CANalyzer (버스 모니터링)
   ├── PREEvision (아키텍처)
   └── DOORS / Jira (요구사항 추적)
```

**우리 프로젝트와 비교**:
| 현업 활동 | 우리 프로젝트 | 상태 |
|---------|------------|------|
| HARA + Safety Goals | H-01~H-03, SG-01/SG-02 | ✅ 완료 |
| System Requirements | Req_001~018 (ASIL 분류 포함) | ✅ 완료 |
| Network Topology (DBC) | vehicle_system.dbc | ✅ 완료 |
| Architecture Design (drawio) | SYS3 7페이지 | ✅ 완료 |
| SW 구현 (CAPL) | nodes/ 7개 노드 명세 | 📋 명세 완료, 코딩 예정 |
| Integration Test | In_Test_01~15 | 📋 명세 완료, 실행 예정 |
| System Test (E2E) | Scene.1~18 | 📋 명세 완료, 실행 예정 |

**결론**: 우리가 가는 방향은 **현업 자동차 SW 개발 프로세스 그 자체**입니다.

## Q6. 프로젝트를 비전공자에게 한 줄로 설명하면?

### 권장 한 줄 요약 (발표/보고서 앞부분에 사용)

> **"자동차 창문 모터 고장이 발생했을 때, 차량 내부 통신망(LIN→CAN→Ethernet)을 거쳐 운전자에게 경고가 울리고, OTA로 소프트웨어가 자동 복구되는 전 과정을 가상 환경(CANoe)에서 검증합니다."**

이 한 줄이 있으면 비전공자도:
- "창문 고장" → 익숙한 상황
- "경고등" → 본 적 있는 결과
- "자동 소프트웨어 업데이트" → Tesla OTA로 알고 있는 개념
- "가상 환경 검증" → 안전하게 테스트한다는 인상

| 항목 | 현재 상태 | 권장 액션 |

| 기술 방향 | ✅ 현업 트렌드(SDV/OTA) 정확히 부합 | 유지 |

| 대분류 적합성 | ⚠️ 표면적으로 UX와 거리 있음 | 발표 시 "편의 인프라" 프레이밍 전략 적용 |
| 가시적 결과물 | ⚠️ 화면 UI 없음 | CANoe 데모 영상 + 파형/로그 보고서 준비 |
| 심사위원 이해도 | ⚠️ 기술 용어 장벽 | 한 줄 요약 + 스토리텔링 발표 구성 |
