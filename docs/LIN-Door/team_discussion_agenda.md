# 팀원 토론 의제 정리 (Team Discussion Agenda)

## 긍정적 의제 (프로젝트 강점 / 지지 의견)

### ✅ 1. SDV/OTA 트렌드 부합 — 시의성 있는 주제 선정
> "현재 차량 아키텍처가 SDV, Zonal 구조로 변화되어 가는 중이라 OTA 업데이트가 중요한 부분이란 생각이 들어..." — 성현

- 현재 완성차/티어1 업계 핵심 화두는 SDV(Software Defined Vehicle) 전환
- OTA(Over-the-Air) 업데이트는 Tesla 이후 전 OEM이 필수 도입 중
- 미래차 SW 개발이라는 대분류와 방향 일치

### ✅ 2. E2E 검증 구조 — 단순 구현이 아닌 시스템 검증 접근
> "Fault Injection부터 Diagnostics, OTA 적용 이후 차량 시스템의 동작 변화까지 End-to-End 관점에서 검증하는 구조" — 성현

- Fault Detection → Gateway Routing → UDS Diagnostics → OTA Update 전 과정을 연결
- 개별 기능 구현이 아닌 시스템 레벨 통합 검증 → 실무 역량 증명 가능
- V-Model 기반 추적성 확보로 체계적 접근 가시화

### ✅ 3. 주제 집중도 향상 — ADAS/동역학 제외, BCM/IVI/OTA 집중
> "저희는 주제랑 연관있는 분야에만 집중하는 것이 좋다고 생각합니다." — 택천
> "차량 전반을 다루는 것 보다는 세부 주제에 집중하는 것이 더 낫다고 생각합니다." — 라엘

- 범위 축소 = 각 기능의 완성도 향상
- Body(BCM+LIN) + Infotainment(Cluster) + OTA 3개 도메인으로 명확화
- 다이어그램, 문서, 코드 모두 핵심 시나리오 중심으로 정렬

### ✅ 4. 도구 활용 구체성 — CANoe 기반 ECU 동작 검증
> "CANoe로 ECU 동작을 검증하는 부분" — 택천

- Vector CANoe는 현업 자동차 SW 개발 표준 도구
- CAPL 스크립팅, DBC 정의, Test Module 등 실무 환경 직접 구현
- SIL(Software-in-the-Loop) 환경에서 실제 HW 없이 검증 가능함을 증명

### ✅ 5. 다이어그램 완성도
> "draw io 확인해보니까 우리가 구현해야 하는 내용들만 정리되어 있는 것 같아서 이대로 가도 좋은 것 같아요" — 라엘

- 7페이지 Concept Design drawio (V-Model 기반 SYS3 설계)
- 아키텍처 → 기능 블록 → 네트워크 토폴로지 → E2E 서비스 흐름 체계적 구성
- 발표 시 시각적 설명 자료로 활용 가능

### ✅ 6. 기술 성장 가치 — 업계 표준 프로세스 경험
- ISO 26262 / ASPICE 기반 문서화 경험 자체가 포트폴리오
- 대기업 면접/실무 투입 시 즉시 활용 가능한 역량
- 단순 코딩 프로젝트와 차별화되는 시스템 엔지니어링 관점

## 부정적 의제 (우려사항 / 검토 필요 사항)

### ⚠️ 1. 대분류 주제 적합성 문제
> "저희 프로젝트 대분류가 '편의 및 UX 시스템'이잖아요. 근데 저희 주제는 고장 진단 OTA 이부분인데... 시스템 인프라에 가까운거 같아서요." — 현준

- 대분류: `CANoe 기반 vECU 시뮬레이션을 통한 미래차 SW 개발 / 편의 및 UX 시스템`
- 고장 진단(DTC) + OTA는 사용자가 직접 체감하는 편의/UX가 아닌 **SW 인프라 영역**
- 평가 기준이 "편의/UX" 에 맞춰져 있다면 주제 미스매치 지적 가능성 존재

### ⚠️ 2. 심사위원 이해도 — 비전공자 대응 어려움
> "심사위원분들은 비전공자가 있다고 멘토님께서 말씀 해주신거 같은데 저희 요구사항을 보고 알수 있을지 걱정이 됩니다." — 현준
> "심사위원분들도 대분류 주제에 따라 평가를 할텐데 그 분들이 이해하기 좋은 방향으로 진행되는 것이 좋지 않을까 합니다." — 택천

- `ASIL-B`, `DTC B1234`, `LIN 0x21`, `UDS 0x19 0x02` 등 전문 용어 남발 시 비전공 심사위원 진입장벽
- "졸음운전 감지 시스템" 같은 직관적 주제 대비 추상적으로 느껴질 수 있음
- 발표/보고서에서 쉬운 언어로 풀어내는 전략 필요

### ⚠️ 3. 가시적 결과물 부재
> "주제 특성 상 가시적으로 나올 게 없어 보인다던 멘토님 말씀이 있으셨으니까 관련해서 질문 드려보면 좋을 것 같습니다." — 라엘

- 모바일 앱, 웹 대시보드 같은 눈에 보이는 결과물 없음
- CANoe 시뮬레이션 화면, 파형, 로그가 결과물이지만 비전공자에게는 낯선 형태
- 심사 시 "이게 뭘 한 건지" 직관적으로 전달하기 어려울 수 있음

### ⚠️ 4. HARA / ISO 26262 포함 필요성 의문
> "안전쪽과 연결짓는 것이 필요한지도 궁금하네요." — 택천
> "ISO랑 HARA도 포함시키는 것이 필요한지도 멘토님께 여쭤보면 좋을것 같습니다." — 택천

- 프로젝트 본질이 "안전 인증 프로세스"인지 "ECU 동작 검증"인지 경계가 모호
- HARA + V-Model 문서 작업량 자체가 큰 부담
- 멘토 확인 필요: ISO 26262 준수가 평가 항목인지, 아니면 참고 수준인지

### ⚠️ 5. 고장 진단 검증 방법론 불명확
> "주제를 고장 진단으로 가버리면 고장 진단에 관한 검증을 어떤식으로 해야할지가 감이 안잡히네요." — 현준

- V-cycle에서 고장 진단을 **무엇으로** 검증하는가? CANoe Test Module? 아니면 별도 도구?
- 검증 결과를 어떻게 수치화하고 보고서에 담는가?
- 합격/불합격 기준(Pass/Fail Criteria)이 명확하지 않으면 테스트 의미 희석

### ⚠️ 6. 주제 범위 및 역할 혼선
> "IVI, BCM도 다룰거라고 하셔서 검증은 부가적으로 들어가는 거라고 이해했는데 맞을까요?" — 라엘
> "저도 완전 검증 쪽으로 주제를 잡는 거라고 생각하진 않고 있었는데." — 라엘

- BCM/IVI 구현이 주(Main)인지, OTA 검증이 주인지 팀 내 인식 차이 존재
- 주제 재정의 및 팀원 간 공동 이해 도출 필요
- 멘토와 방향 재확인 권장
