# 멘토 샘플 분석 및 현재 방향성 비교

## 📊 멘토 샘플의 V-모델 구조 분석

### 전체 구조 (13개 시트)

```
┌─────────────────────────────────────────────────┐
│ V-Model 기반 산출물 구조                          │
└─────────────────────────────────────────────────┘

1. 일정표 (Schedule)
   └─ 프로젝트 단계별 계획/수행 일정

2. 요구사항 정의 단계 ─┐
   └─ 01_Requirements    │
                         │
3. 시스템 설계 단계 ─────┤
   ├─ 02_Concept_design  │
   └─ 03_Function definition │
                         │
4. 상세 설계 단계 ───────┤
   ├─ 0301_SysFuncAnalysis         │ V-Model 좌측
   ├─ 0302_NWflowDef               │ (설계)
   ├─ 0303_Communication Specification │
   └─ 0304_System Variables        │
                         │
5. 구현 단계 ────────────┤
   └─ 04_System implementation     │
                         │
6. 검증 단계 ────────────┘
   ├─ 05_Unit Test                 │ V-Model 우측
   ├─ 06_Integration Test          │ (검증)
   └─ 07_System Test               │

13. messages (부가 정보)
```

## 🔍 각 시트별 상세 분석

### 1️⃣ 일정표
- **구성**: 프로젝트 개발환경, 설계, 구현, 테스트, 최종 산출물
- **특징**: 계획/수행 일정을 간트 차트 형식으로 관리

### 2️⃣ 01_Requirements (요구사항 명세서)

**컬럼 구조:**
```
Req. ID | 요약 | 설명 | 중요도/긴급도 | 요청자 | 검토자 | 변경사항
```

**특징:**
- ✅ **간결한 요구사항**: Req_001 ~ Req_020 (총 20개)
- ✅ **기능 중심**: "무엇을"만 명세 ("어떻게"는 없음)
- ✅ **중요도 표시**: 상/상, 상/중 등 우선순위
- ✅ **카테고리 분류**:
  - 운전자용 제어 기능 (Req_001~007)
  - 운전자 졸음에 의한 자동 제어 기능 (Req_008~017)
  - 휴식 권장 메시지 (Req_018~020)

**예시:**
```
Req_001: 시동 버튼의 Default 상태는 off 이다.
Req_002: 시동 버튼을 누르면 on 상태가 되고, 모든 ECU가 활성화...
```

⚠️ **멘토 피드백과의 연관성:**
- "요구사항을 너무 디테일하게 처음부터 들어가셨어요"
- "샘플 엑셀을 거의 따라서 작성하세요"
- → 샘플은 **레이턴시, 성능 지표 등을 요구사항에 포함하지 않음**

---

### 3️⃣ 02_Concept_design (컨셉 디자인)

**특징:**
- 📊 **비어있음** (이미지/다이어그램으로 구성된 것으로 추정)
- 멘토가 "이번 주에 제출하라"고 한 부분:
  - ✅ 컨셉 디자인
  - ✅ 네트워크 구성
  - ✅ 서비스 목록
  - ✅ 구성 요소

**멘토 요구사항:**
- "수작업으로 그리기 (PPT 또는 Visio)"
- "AI 도구 사용 자제"
- "의미 파악에 집중"

---

### 4️⃣ 03_Function definition (기능 정의)

**구조:**
```
가상노드(Simulator) | 분류 | 기능명 | 기능설명 | 비고 | 검증
```

**카테고리:**
1. **입력**: 시동 스위치, 운전자 상태, CO2 수치, 브레이크/엑셀 페달, 비상등 버튼, 기어 버튼
2. **출력**: RPM, 속도, CO2, 기어 상태, 비상등, 창문 상태, 운전자 운행 정보
3. **ECU 동작**:
   - 엑셀 ECU
   - 브레이크 ECU
   - CO2 센서 ECU
   - 클러스터 ECU
   - DSW ECU
   - Gateway
   - 핸들 ECU
   - 비상등 ECU
   - 변속기 ECU
   - 창문 ECU
   - 엔진 ECU

**특징:**
- ✅ **입력부, 제어부, 출력부 명확히 구분**
- ✅ **각 ECU의 역할 간결하게 정의**
- ✅ **구현 방법도 간단히 명시** (예: "Timer 활용", "CAPL 활용 메시지 복사")

---

### 5️⃣ 0301_SysFuncAnalysis (시스템 기능 분석)

**구조:**
```
노드 | 기능 상세 | 비고
```

**도메인별 분류:**
1. **Powertrain**
   - Transmission Ctrl
   - Engine Ctrl

2. **Chassis**
   - Brake Ctrl
   - Handle Ctrl
   - Acceleration Ctrl

3. **Body**
   - Emergency Lights Ctrl
   - Window Ctrl
   - CO2 Meter Ctrl
   - DSW Ctrl

4. **Infotainment**
   - Cluster Ctrl

5. **Actual Device** (실제 장치)
   - Emergency Lights
   - Cluster
   - Transmission
   - Acceleration Pedal
   - Brake Pedal
   - Window

**특징:**
- ✅ **도메인별 그룹화**
- ✅ **Ctrl(제어기)와 Device(실제 장치) 분리**
- ✅ **기능 상세는 1-2줄로 간결하게 작성**

---

### 6️⃣ 0302_NWflowDef (네트워크 흐름 정의)

**구조:**
```
Channel | ID hex | Symbolic Name | Byte no. | Function | Bit no. | signal name |
AccelCtrl | BrakeCtrl | ClusterCtrl | ... (각 ECU별) | Gateway | [비고]
```

**특징:**
- ✅ **DBC 파일과 유사한 구조**
- ✅ **각 ECU별 Tx/Rx 표시**
- ✅ **메시지 흐름을 테이블로 정리**

**예시:**
```
Chassis | 0x100 | frmBrakeMsg | 0 | Input Brake rate | 0 | BrakeAngle
         BrakeCtrl: Tx, EngineCtrl: Rx
```

⚠️ **멘토 피드백:**
- "DBC가 너무 먼저 나왔어요"
- → **올바른 순서**: 기능 정의 완료 → 통신 구성 완료 → 시스템 아키텍처 완료 → DBC

---

### 7️⃣ 0303_Communication Specification (통신 명세)

**구조:**
```
Message | Identifier | DLC | Signal | signal bit position |
Data 설명 | Data 범위 | Data 사용
```

**특징:**
- ✅ **DBC 파일의 "사람이 읽을 수 있는" 버전**
- ✅ **각 신호의 목적과 범위 명확히 정의**

**예시:**
```
frmBrakeMsg | 0x100 | 1 | BrakeAngle | 0~4 |
현재 브레이크 패달 각도 | 0~31 | 현재 브레이크 패달 각도를 EngineECU로 전송
```

---

### 8️⃣ 0304_System Variables (시스템 변수)

**구조:**
```
ID | Namespace | Name | Data type | Min | Max | Initial Value | Description
```

**특징:**
- ✅ **CANoe 시스템 변수 직접 정의**
- ✅ **Namespace로 도메인 구분** (Chassis, Body, Engine, Button, etc.)

**예시:**
```
1 | Chassis | accelAngle | Double | 0 | 30 | 0 | 액셀 페달 각도
5 | IGN | IgnStart | uint32 | 0 | 1 | 0 | 시동 여부 판단
```

### 9️⃣ 05_Unit Test (단위 테스트)

**구조:**
```
노드 | 분류 | 기능명 | 기능 설명 | Pass/Fail | 담당자 | 일자
```

**특징:**
- ✅ **각 ECU/노드별 개별 기능 검증**
- ✅ **가상 노드(Simulator)의 입출력도 테스트**

---

### 🔟 06_Integration Test (통합 테스트)

**구조:**
```
테스트 ID | 요구사항 ID | 테스트 목적 | 예상 결과 |
테스트 수행 결과 | 담당자 | 일자
```

**특징:**
- ✅ **요구사항과 1:1 또는 1:N 매핑**
- ✅ **여러 ECU가 협력하는 시나리오 검증**

**예시:**
```
In_Test 1 | Req_001, Req_002 | 시동 버튼 Off 일 때 ECU Deactivate 확인 |
시동 껐을 때 ECU 동작하지 않음 | Pass
```

⚠️ **V-Model 특징:**
- 요구사항(좌측) ↔ 통합 테스트(우측) 대칭 구조

---

### 1️⃣1️⃣ 07_System Test (시스템 테스트)

**구조:**
```
Scene. ID | 설명 | Pass / Fail | 담당자 | 일자
```

**특징:**
- ✅ **실제 사용 시나리오 기반**
- ✅ **Scene 1~20까지 순차적 진행**

**예시:**
```
Scene. 1: Power OFF 상태로 테스팅 시작
Scene. 2: Power ON으로 변경된 후 기어 상태가 P인지 확인
Scene. 10: 졸음 버튼을 통해 깨어있음 -> 졸음으로 변경
```

## 🔄 우리 팀 vs 멘토 샘플 비교

### ❌ 우리 팀의 문제점

| 항목 | 우리 팀 | 멘토 샘플 | 멘토 피드백 |
|------|---------|-----------|------------|
| **요구사항 명세** | 과도하게 상세함<br>(레이턴시, 성능 지표 포함) | 간결하고 기능 중심<br>(20개 요구사항) | "요구사항을 너무 디테일하게 처음부터 들어가셨어요" |
| **아키텍처 다이어그램** | ECU → Gateway 개별 연결<br>(이더넷 방식) | CAN 버스 구조<br>(한 줄로 연결) | "이렇게 그리신 거는 이더넷이에요" |
| **도메인 아키텍처** | "CAN High Speed로 묶인다"<br>(통신 개념 혼재) | 도메인 간 데이터 교환만 표현 | "도메인별로 주고받는 데이터만 표현" |
| **DBC 파일** | 요구사항 단계에서 먼저 작성 | 시스템 아키텍처 완료 후 작성 | "DBC가 너무 먼저 나왔어요" |
| **개발 순서** | 애자일 방식<br>(빠른 구현 → 문서 보완) | V-모델<br>(문서 완성 → 구현) | "차량 시스템은 애자일 방식과 적합하지 않음" |
| **CANoe 위치** | 아키텍처 다이어그램에 포함 | 도구일 뿐<br>(다이어그램에 미포함) | "CANoe는 환경 구축 도구" |

---

### ✅ 우리 팀의 강점

| 항목 | 평가 |
|------|------|
| **자료 조사** | ✅ OpenDBC, 47개 ECU 조사 완료 |
| **정리 능력** | ✅ "정리를 엄청 잘하셨네요" (멘토 피드백) |
| **학습 의지** | ✅ 적극적인 질문과 빠른 실행력 |
| **도메인 분류** | ✅ Powertrain, ADAS, Body, Infotainment 분류 완료 |

## 🎯 멘토가 원하는 방향성 명확화

### 1️⃣ V-사이클 기반 프로세스

```
┌──────────────────────────────────────────────────────────────┐
│                    V-Model Process                            │
└──────────────────────────────────────────────────────────────┘

Week 2 (현재)
  └─ 프로젝트 주제 재정립, 차량 시스템 이해

Week 3 (다음 주)
  ├─ 요구사항 도출
  ├─ 기능 정의
  └─ 컨셉 디자인

Week 4
  ├─ CANoe 준비 완료
  ├─ 데이터베이스 완성
  ├─ 네트워크 설계 및 통신 흐름 명세 완료
  └─ ⚠️ 문서 완료 마감

Week 5 이후
  ├─ 구현 시작
  ├─ 테스팅
  └─ 구현 최적화

Week 7
  └─ 발표 자료 최종 제출
```

### 2️⃣ 문서 작성 철학

> **"문서를 먼저 끝내는 것보다 개발 스킬을 사이사이 올려두세요"**

- ✅ 문서는 천천히 **확실하게**
- ✅ CANoe 트레이닝 병행
- ❌ 문서 빨리 끝낸다고 프로젝트 빨리 끝나는 게 아님
- ❌ 문서가 확실하지 않으면 2~3번 개발해야 함

### 3️⃣ 아키텍처 레벨 구조

```
Level 1: 차량 전체 시스템
  └─ CAN 버스 구조 (계층적 표현)
  └─ Central Gateway 중심

Level 2: 도메인별
  └─ Powertrain, Chassis, Body, Infotainment
  └─ 도메인 간 데이터 교환만 표현 (통신 X)

Level 3: 통신 구조
  └─ CAN 네트워크
  └─ 데이터 흐름 (시퀀스 다이어그램)

Level 4: 특화 ECU (IVI)
  └─ Level 1~3 전부 구현
  └─ Level 4는 1~2개 특화 (IVI 상세 구현)
```

## 💡 핵심 인사이트

### 방향 전환의 필요성

| Before (웹 개발 방식) | After (임베디드 개발 방식) |
|----------------------|--------------------------|
| 애자일 | V-모델 |
| 빠른 개발 | 확실한 문서화 |
| AI 의존 | 수작업 이해 |
| 작은 규모로 시작 → 확장 | 처음부터 전체 정의 → 구현 |

### 산출물 ↔ 개발물 일치

> **"산출물을 개발 코드로 투영했을 때 맞아떨어져야 함"**
> **"개발물을 산출물로 만들었을 때도 맞아떨어져야 함"**

```
요구사항 명세서
    ↓
기능 정의
    ↓
```

## 📚 참고: 멘토 샘플의 핵심 메시지

> **"차량 시스템 개발 ≠ 애자일"**
>
> "차량 시스템은 애자일 방식과 적합하지 않음. 규모를 작게 해서 키워나가는 방식이 아님. V-모델: 정의 → 구현 → 정의 → 구현 (하브 꼭지). 처음부터 정의, 구현 내용을 확실하게."

> **"문서 vs 개발"**
>
> "문서 빨리 끝낸다고 프로젝트 빨리 끝나는 게 아님. 문서가 확실하지 않으면 2~3번 개발해야 함. 문서는 늦어도 괜찮지만 확실하게. 개발 스킬은 병행해서 올려두기."

> **"산출물 vs 개발물 일치"**
>
> "산출물을 개발 코드로 투영했을 때 맞아떨어져야 함. 개발물을 산출물로 만들었을 때도 맞아떨어져야 함. 역으로 뽑든 정으로 뽑든 상관없지만 일치해야 함."
