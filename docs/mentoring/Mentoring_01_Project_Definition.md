# 프로젝트 정의서 (1-Page)

## 📌 주제 (한 문장)
**CANoe 기반 차량 분산 시스템 아키텍처 설계 및 ECU 간 통신 검증**

---

## 🎯 명확한 문제 정의

### 현재 상황
- 차량은 **다수의 ECU**(엔진, 변속기, 제동, 조향, IVI 등)가 **CAN 네트워크**로 연결된 **분산 시스템**
- 각 ECU는 독립적으로 동작하지만, 차량 전체 서비스를 위해 **데이터 공유 필수**
- 예: 엔진 ECU의 속도 정보 → 클러스터/IVI 표시, ADAS 센서 경고 → IVI 시각화

### 문제점
1. **차량 시스템 전체 구조에 대한 이해 부족**
   - ECU 종류, 역할, 도메인 분류를 모름
   - 왜 통신이 필요한지 설명 못함

2. **실제 개발 환경 부재**
   - 실제 차량 없이 시스템 검증 불가능
   - 하드웨어 비용 및 안전 문제

3. **통합 테스트 어려움**
   - 다수 ECU 동시 동작 검증 필요
   - 결함 주입 및 시나리오 테스트 필요

---

## 💡 문제 해결 방안

### 1. 차량 시스템 아키텍처 설계 (Level 1-3)
**Level 1: 차량 전체**
- 4개 도메인 정의: Powertrain, Chassis, Body, Infotainment
- 15+ ECU 배치 및 역할 정의
- CAN 네트워크 구조 설계

**Level 2: 도메인별 상세**
- 각 도메인 내 ECU 구성
- ECU 간 데이터 흐름

**Level 3: CAN 통신 구조**
- CAN 데이터베이스 (.dbc) 설계
- 메시지 ID 할당 (0x000-0x7FF)
- 시그널 정의 (이름, 길이, 단위, 범위)

### 2. CANoe 가상 차량 환경 구축
- **CANoe = 가상 차량 시뮬레이터**
- 실제 차량 없이 모든 ECU 동작 재현
- CAPL 스크립트로 각 ECU 구현
- 통신 검증 및 결함 주입 테스트

### 3. IVI ECU 특화 기능 구현 (Level 4)
- 앰비언트 라이팅 제어 (IVI → BCM)
- ADAS 경고 표시 (Camera ECU → IVI)
- 테마/프로필 관리
- **기존 작업물 활용** (ADAS 통합, UI 아키텍처)

## 📦 산출물

### 1. 시스템 아키텍처 문서
- **Level 1**: 차량 전체 시스템 아키텍처 (PlantUML)
- **Level 2**: 도메인별 상세 아키텍처 (4개 파일)
- **Level 3**: CAN 통신 구조 다이어그램 (2개 파일)
- **Level 4**: IVI ECU 상세 설계 (선택적, 기존 작업 활용)

### 2. CAN 데이터베이스
- **vehicle_system.dbc**: 전체 차량 CAN 메시지 정의
  - 메시지 ID: 0x000-0x7FF
  - 시그널: 100+ 개 (속도, RPM, 조향각, ADAS 이벤트 등)
  - 네트워크 노드: 15+ ECU

### 3. CANoe 프로젝트
- **가상 차량 환경**: 모든 ECU를 CAPL 노드로 구현
- **테스트 케이스**: 시나리오 기반 통신 검증
- **결함 주입**: 메시지 drop, timeout, invalid value 테스트

### 4. IVI 특화 기능
- 앰비언트 라이팅 제어
- ADAS 경고 표시 (LDW, AEB, BSD)
- 테마/프로필 관리

### 5. 테스트 리포트
- 통신 검증 결과
- 결함 주입 테스트 결과
- 성능 측정 (응답 시간, 메시지 전송률)
