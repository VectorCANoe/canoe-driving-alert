# CANoe 개발기 2편: 다시 설계하라고 했는데, 그게 진짜 시작이었다

> 1편에서 우리는 스타 토폴로지 그림을 들고 갔다가 두 번 얻어맞고 돌아왔다. 이번엔 그 다음 이야기다.

---

## 다시 시작한다는 게, 진짜로 처음부터라는 뜻이었다

멘토링 피드백을 정리하면서 팀 내에서 나온 말이 있다.

> "우리가 기초부터 다시 해야 하네."

이 말이 나왔을 때 솔직히 좀 막막했다. 이미 PlantUML 다이어그램 11개, DBC 파일, 147개 메시지 문서까지 만들었는데 그걸 다 엎어야 한다는 거였으니까. 근데 실제로 엎어야 했다.

멘토링 6회차에서 정리된 핵심 진단이 이거였다.

> **"우리는 지붕부터 올리고 있었다."**

기능이 뭔지도 정의 안 한 상태에서 DBC 파일 만들고, 시스템 아키텍처도 안 잡혔는데 CAPL 코드 짜고 있었다. 순서가 완전히 거꾸로였다.

---

## Top-Down으로 접근한다는 게 어떤 뜻인지

이전까지 팀이 했던 방식은 이랬다.

```
"이런 기능 있으면 좋겠다" → 메시지 어떻게 보낼까 → DBC 만들자 → 구현하자
```

V-Model에서 요구하는 방식은 다음과 같다.

```
전체 시스템은 뭔가? → 도메인별 기능은? → 도메인 간 어떤 데이터가 오가나? → (그 이후에) 메시지가 뭔가?
```

멘토링 가이드 문서에 적혀있던 표현이 정확했다.

> **"여기서 'CAN 메시지 ID 0x123을 보낸다'라고 쓰면 안 됩니다. 'RPM 정보를 보낸다'와 같이 데이터의 흐름 관점에서 서술하세요."**

우리는 항상 ID 먼저 정의하고 있었다. 그게 문제였던 거다.

## CAN 버스 토폴로지, 이번엔 제대로 다시 그렸다

멘토링 7회차에서 실제로 멘토님이 그림을 그려가며 설명하셨다. 회의 녹취를 들으면 이런 내용이 나온다.

> "제가 선 바꾼 거 보시면 이게 끝이에요. 여기서 더 고민할 게 없어요."

우리가 헷갈렸던 핵심이 뭐냐면, 이더넷과 CAN을 같은 선 위에 올려놓은 거였다. 백본 네트워크로 이더넷을 깔았으면 도메인 게이트웨이 아래를 CAN으로 연결하면 되는 건데, 우리는 "이더넷도 CAN도 다 깔아놨으니까 이중화가 되는 거 아닌가?" 라는 착각을 했다.

멘토님 말씀이 명쾌했다.

> "그랜저를 타고 회사 가는데 혹시나 그랜저가 나빠질 수도 있으니까 자전거를 차 위에 올려두겠다는 말이에요. 자전거로 1시간 걸리는 거리를 가면 지각이잖아요. 어차피 서비스는 끝난 거예요."

이더넷이 백본인데 CAN FD를 이중화라고 놓는 건, 자전거를 이중화 수단이라고 부르는 것과 같다. 완전히 틀린 논리였다.

올바른 구조가 이렇다는 걸 직접 그려서 확인했다.

![올바른 CAN 버스 토폴로지: Ethernet Backbone + 도메인별 CAN](/Users/juns/code/work/mobis/PBL/driving-alert-workproducts/02_network.png)

게이트웨이는 네트워크 간 프로토콜을 변환하는 역할이고, 같은 도메인 안 ECU끼리는 CAN Bus 하나로 묶인다. 이 원칙 하나만 제대로 이해하면 그림이 바로 잡힌다.

## 입력-제어-출력, 이게 왜 중요한가

우리가 처음 그린 컨셉 디자인이다.

![컨셉 디자인 초안: 입력부/제어부/출력부 구조](/Users/juns/code/work/mobis/PBL/driving-alert-workproducts/02_concept.png)

멘토링에서 이 그림을 보여드렸을 때 다음과 같은 피드백이 나왔다.

> "핸들은 출력일까요, 제어일까요, 입력일까요?"

팀원이 "출력에 가깝다"고 답하자 멘토님이 "그러면 지워주시면 돼요."라고 하셨다.

이 질문 하나가 핵심을 짚은 거였다. 컨셉 디자인에서 다뤄야 할 것은 **사용자가 무언가를 서비스 요청하거나 제어했을 때, 어떤 과정을 거쳐서 결과가 어떻게 나오는가**의 흐름이다. 중간에 내부적으로 제어당하는 장치들(핸들, 모터)은 그 흐름 안에 있는 거지, 별도로 출력부에 그려야 하는 대상이 아니다.

예시를 들면 이렇다.

```
입력: 엑셀 페달
제어: 엔진 ECU (토크 조절)
출력: 계기판 속도 표시
```

```
입력: 고속도로 주행 중 핸들 조작 없음 (입력값 = 0)
제어: ADAS_WARN_CTRL (차선 이탈 감지 로직)
출력: 앰비언트 조명 주황색 경고
```

우리가 그렸던 컨셉 디자인은 이 흐름이 명확하지 않아서 핸들이 제어부에 있기도 하고 출력부에 있기도 했다. 그걸 정리하는 게 이 단계의 핵심이었다.

## 요구사항을 다이어트 너무 많이 했다

7회차 멘토링에서 나온 말 중에 제일 기억에 남는 게 이거다.

> "다이어트를 너무 많이 하셨어요."

우리 요구사항 명세서가 10개 남짓이었는데, 멘토님 말씀으로는 최소 요구사항 10개에 노드 10개, 그 사이를 흐르는 메시지가 100개 가까이 있어야 포트폴리오로 쓸 수 있는 수준이 된다는 거였다.

우리가 뭘 잘못했냐면, "우리 핵심 시나리오에 쓰이는 것만" 골라내다 보니까 기본 차량 기능이 통째로 빠진 거였다. 경찰차가 신호를 보내는 시나리오는 있는데, 그 차가 어떻게 달리는지는 아예 없었다. 브레이크, 엔진, 조향, 창문 같은 기본 기능이 하나도 없는 차였던 거다.

> "차량이라는 게 좀 있어야 되는데, 그냥 애플리케이션 하나니까."

맞는 말이었다. 차량 회사 포트폴리오에서 '차량 시스템을 이해했다'고 어필하려면, 실제로 차 한 대가 굴러가는 구조를 구현해야 하는 거였다.

## DBC 파일도 다시 짜야 했다

8회차 멘토링에서 DBC 파일 피드백이 나왔다.

> "한 네트워크에 게이트웨이가 두 개 있을 수는 없어요. 네트워크당 게이트웨이는 하나예요."

우리가 만든 DBC는 모든 걸 하나의 파일에 때려넣은 상태였다. 멘토님 설명으로는 **네트워크마다 DBC 파일 하나**가 맞는 구조다.

비유가 이거였다.

> "1반 2반 3반이 있어요. 1반 안에 있는 사람들 간의 대화 내용을 정의한 게 1반 DBC. 반장들이 복도에서 얘기하는 게 게이트웨이 간 DBC. 2반 반장은 1반에 엔진이라는 녀석이 있는지 몰라요. 반장 입으로부터 엔진 데이터를 듣는 거죠."

그리고 이더넷은 DBC로 만들 수 없다. DBC는 CAN 네트워크 전용 포맷이기 때문에, 이더넷 통신은 별도의 메시지 정의 표(엑셀)로 관리해야 한다는 것도 알게 됐다.

## 단위-통합-시스템 테스트를 왜 나누나

8회차 멘토링에서 V-Model의 테스트 단계에 대해서도 얘기가 나왔다.

팀원 중 하나가 "시나리오 테스트 하나만 하면 되지 왜 단위/통합/시스템을 나누냐"는 질문을 했다. 멘토님 답변이 이랬다.

> "오래 걸리니까 쪼갠 거예요."

차량 SW는 수십 개 회사가 ECU를 나눠서 만들고, 그게 다 합쳐져서 자동차 한 대가 된다. 그걸 다 모아서 처음으로 시동 거는 순간 테스트하면, 문제가 생겼을 때 어느 회사 어는 ECU 얘기인지도 모른다. 그러면 내일 출시해야 하는 차가 지금 멈춰있는 거다.

작게 검증하면서 올라가는 이유는 **복잡한 시스템이 완성되기까지 오래 걸리기 때문**이다. 미리 작은 단위에서 잡아야 나중에 큰 사고가 안 난다.

> "게임은 버그 나면 롤백하면 되죠. 차는요. 사람이 이제 갑니다."

차량 SW에서 문제가 생기면 사람이 다치거나 죽는다. 그래서 작게 검증하면서 올라가는 거다. 롤백이 없는 시스템이기 때문에.

## 2편을 마치며

1편에서 두 번 방향이 틀렸다고 맞았고, 2편에서는 그 방향을 맞추는 과정에서 계속 새로운 걸 모른다는 게 드러났다.

그래도 이 시점에서 팀이 이해하게 된 게 있다.

- V-Model은 형식이 아니라 이유가 있는 구조다
- Top-Down으로 설계한다는 건, 시스템 전체를 먼저 보고 내려오는 거다
- 차량 SW가 빡빡한 건 사람이 타기 때문이다
- 요구사항이 빈약하면 구현도, 테스트도, 포트폴리오도 다 빈약해진다

3편에서는 이 모든 걸 반영해서 실제로 시나리오를 확정하고, 주제를 "주행 상황 실시간 경고 시스템"으로 피벗한 이후의 이야기를 쓸 예정이다. 프로젝트가 처음으로 '이제 뭘 만들어야 할지 알겠다' 싶었던 순간이다.
