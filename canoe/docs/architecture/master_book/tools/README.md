# Master Book Tools

Subtitle: Tracked build and render steps for the official master-book asset pack.

These tracked scripts regenerate the official `master_book` asset pack in place.

## Files

- `build_master_book_asset_pack.py`
  - builds the ECU metadata dataset, grouped SVG figures, the official `FLOW_01~20` action-flow summary SVG pack, the detailed signal-flow triplet pack, and 101 ECU cards
- `render_master_book_pdf.py`
  - converts the generated markdown master book into HTML and PDF

## Standard Order

1. `python tools/build_master_book_asset_pack.py`
2. `python tools/render_master_book_pdf.py`

## Output Root

All generated outputs land in `canoe/docs/architecture/master_book/**`.

## Build Ownership

- `build_master_book_asset_pack.py` is the official tracked build step for both flow layers.
  - action summaries: `svg/flows/action/FLOW_01~20_*.svg`
  - detailed signal triplets: `flows/signal/*.puml`, `svg/flows/signal/*.svg`, `png/flows/signal/*.png`
  - indexes: `ACTION_FLOW_INDEX_2026-03-28.md`, `SIGNAL_FLOW_INDEX_2026-03-28.md`
- `render_master_book_pdf.py` consumes the generated master-book outputs after the asset pack exists.
  - it does not define a separate unofficial flow-render path
  - if flow assets change, rerun `build_master_book_asset_pack.py` first

## Flow Layer Note

- `svg/flows/action/FLOW_01~20_*.svg`
  - official action-flow summary layer generated by the standard build
- `flows/signal/*.puml` + `svg/flows/signal/<signal-flow>.svg` + `png/flows/signal/*.png`
  - detailed signal-flow triplet layer kept alongside the summary pack for drill-down and appendix reuse

## Publication Rule

- The default master-book body publishes the `20` action-flow summaries first.
- The detailed signal triplets stay as the official appendix/drill-down pack.
- Promote detailed signal triplets into reviewer-facing chapters only selectively, by group or ECU need, instead of bulk-inlining all triplets into the main body.

Archived prototype studies now live under `canoe/docs/architecture/master_book/prototypes/card_prototypes/`.
They are preserved for layout reference only and are not used by the standard build/render flow.
